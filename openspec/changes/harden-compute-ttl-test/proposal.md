## Why

The [`ComputeTtlToNextResetTest`](../../../core/domain/src/test/kotlin/id/nearyou/app/core/domain/ratelimit/ComputeTtlToNextResetTest.kt) "Different users produce different offsets at high probability" test asserts `differingPairs >= 999` over 1000 random `UUID` pairs ([test:55-71](../../../core/domain/src/test/kotlin/id/nearyou/app/core/domain/ratelimit/ComputeTtlToNextResetTest.kt)) — but the math is over-specified relative to achievable randomness. Per-pair collision probability is `1/3600` (the `hashCode().toLong().absoluteValue % 3600L` offset bucket per [`ComputeTtlToNextReset.kt:36`](../../../core/domain/src/main/kotlin/id/nearyou/app/core/domain/ratelimit/ComputeTtlToNextReset.kt)); expected collisions in 1000 pairs ≈ 0.278 (Poisson), so P(≥2 collisions) ≈ 3.2% via Poisson approximation. The test has a baked-in ~3-4% flake rate even on a perfect hash distribution. Confirmed empirically: 3 flakes during the `chat-foundation` ship cycle alone (CI run 25132387140 attempts 1, 2, 3; passed on attempt 4 — see [`FOLLOW_UPS.md` § compute-ttl-to-next-reset-test-flake](../../../FOLLOW_UPS.md)). Each flake costs a manual retry click + ~5-7 min CI runtime per occurrence; the 2026-05-09 `/triage-follow-ups` sweep promoted this entry to `/next-change` as the single OpenSpec hand-off of that cycle.

The matching spec scenario at [`openspec/specs/rate-limit-infrastructure/spec.md:163-165`](../../specs/rate-limit-infrastructure/spec.md) ("for at least 999 of 1000 random pairs, the returned `Duration` values differ") encodes the same over-tight threshold — the spec is the canonical source of the test's tight bound, not the other way around. Without realigning the spec, future test refactors would re-derive the same flaky 999/1000 contract from the spec.

## What Changes

- **MODIFIED capability**: `rate-limit-infrastructure` — RESTATE the existing scenario "Different users different offsets (with high probability)" inside the `computeTTLToNextReset(userId)` shared helper requirement. Bump the random-pair sample from 1000 to 100,000 and the differing-pair threshold from "at least 999" to "at least 99,500" (≥ 99.5%). New threshold preserves the random-pair semantics (the property under test is still "different users produce different offsets with overwhelming probability") and tightens the percentage-of-bucket-coverage promise (99.5% vs the original implicit 99.9%) while making the assertion safely above any plausible Poisson-noise threshold. Math: with collision rate `λ = 100,000 × (1/3600) ≈ 27.78` and `σ ≈ 5.27`, the threshold "≥ 99,500 differing" = "≤ 500 collisions" sits ~89σ above the expected mean, so failure probability is effectively zero (<<10⁻¹⁰⁰). The other six scenarios in the requirement are unchanged.
- **Implementation**: update [`ComputeTtlToNextResetTest`](../../../core/domain/src/test/kotlin/id/nearyou/app/core/domain/ratelimit/ComputeTtlToNextResetTest.kt) "Different users produce different offsets at high probability" — change `repeat(1000)` to `repeat(100_000)` and `(differingPairs >= 999) shouldBe true` to `(differingPairs >= 99_500) shouldBe true`. Update the trailing inline comment that explains the math (currently misdiagnoses the flake source as "hashCode distribution" — the assertion threshold is the actual culprit). Verify total runtime of the affected test stays under ~5s on the CI runner (current ~50ms × 100 ≈ 5s upper bound; UUID generation + ZoneId conversion are the cost dominants).
- **Tests**: no new test classes. Single-file edit to the existing test method.

No new capability is introduced. No schema, migration, REST API, Detekt rule, or runtime behavior changes. Approach 3 (sample-size scaling) is selected over Approach 1 (seeded RNG) and Approach 2 (loosen-only spec text) per the FOLLOW_UPS recommendation — the rationale is in `design.md`. The `:core:domain` source-set choice (the test currently lives at `src/test/kotlin/...` per the JVM-only test source-set, NOT `src/commonTest/kotlin/...` as the parent capability spec scenario "Helper lives in :core:domain" mentions in passing) is preserved as-is — the source-set discrepancy is pre-existing and out of scope.
