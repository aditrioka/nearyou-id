## Why

Notifications are written to the `notifications` table by `NotificationService.emit()` (shipped in [`in-app-notifications`](../../specs/in-app-notifications/spec.md)) but no FCM push is sent — the only `NotificationDispatcher` implementation today is a no-op `InAppOnlyDispatcher` that logs at INFO and goes nowhere. The `user_fcm_tokens` schema and `POST /api/v1/user/fcm-token` endpoint shipped in [`fcm-token-registration`](../../specs/fcm-token-registration/spec.md) (PR [#55](https://github.com/aditrioka/nearyou-id/pull/55)) collect tokens that are currently never used. Without an FCM dispatch path, every "you got a like / reply / follow" event reaches the user only when they next open the app — fine for low-frequency events, broken for chat (which lands in Phase 2 next). This change closes that loop end-to-end and resolves the explicit forward-references already booked into the two predecessor specs ([`in-app-notifications/spec.md` "NotificationDispatcher seam for future FCM"](../../specs/in-app-notifications/spec.md) and the deferred on-send-failure prune contract documented at [`docs/05-Implementation.md:1399`](../../../docs/05-Implementation.md), narrowed in this change's `design.md` D6 to `UNREGISTERED` + `SENDER_ID_MISMATCH` only).

Roadmap anchor: [`docs/08-Roadmap-Risk.md`](../../../docs/08-Roadmap-Risk.md) Phase 2 items 7 ("FCM push infrastructure") and 8 ("Platform-specific push payload builder: Android data-only, iOS alert + mutable-content + NSE handshake via App Group"). Canonical platform-delivery contract at [`docs/04-Architecture.md § Push Notification Infrastructure`](../../../docs/04-Architecture.md) (lines 502–558).

## What Changes

- **NEW** `:infra:fcm` Gradle module wrapping the Firebase Admin SDK (server-side). Service-account JSON read via the `secretKey(env, name)` helper — `firebase-admin-sa` in prod, `staging-firebase-admin-sa` in staging (slot already reserved per [`docs/08-Roadmap-Risk.md:51`](../../../docs/08-Roadmap-Risk.md)). No FCM SDK import outside `:infra:fcm` (`CLAUDE.md` invariant: "No vendor SDK import outside `:infra:*`"; the `NotificationDispatcher` interface stays in its current `:core:data` location, and `:backend:ktor` only sees the interface).
- **NEW** `FcmDispatcher` implementation of the existing `NotificationDispatcher` interface, living in `:infra:fcm`. On each emitted notification, after the DB commit, it: (a) atomically reads the recipient's active tokens AND captures `dispatchStartedAt: Instant` via `SELECT NOW()` on the same DB connection per `design.md` D12 (using the DB clock domain — NOT JVM `Instant.now()` — eliminates JVM↔DB clock-skew false-deletes); (b) looks up the actor's username via `ActorUsernameLookup` reading from `visible_users` (NOT raw `users`) so shadow-banned and deleted actors mask to `null` per `design.md` D13; (c) builds platform-specific payloads via per-platform builders; (d) dispatches in parallel per device via the FCM Admin SDK on a `Dispatchers.IO.limitedParallelism(8)` scope; (e) on FCM `MessagingErrorCode.UNREGISTERED` OR `SENDER_ID_MISMATCH` for a specific token, invokes `deleteTokenIfStale(userId, platform, token, dispatchStartedAt)` which DELETEs the row only if `last_seen_at <= :dispatch_started_at` (preserves freshly-re-registered rows under the race); (f) on `INVALID_ARGUMENT` (which is overloaded by the FCM Admin SDK between stale-token and oversized-payload — see `design.md` D6), `QUOTA_EXCEEDED`, `UNAVAILABLE`, `INTERNAL`, or any non-Firebase exception (network, timeout, deserialize), logs structured WARN and does NOT delete the token (matches the "silent drop, user sees it on next app open via the in-app `notifications` list" fallback at [`docs/04-Architecture.md:558`](../../../docs/04-Architecture.md)).
- **NEW** Per-platform payload builders in `:infra:fcm`:
  - **Android** — data-only message (no `notification` block), `priority: "high"`, fields `type`, `actor_user_id`, `target_type`, `target_id`, and a `body_data` JSON-stringified blob. Per [`docs/04-Architecture.md:506`](../../../docs/04-Architecture.md) ("data-only FCM messages. App handles rendering locally with preference check. High priority").
  - **iOS** — alert push with title + body in the APNs payload, `apns.payload.aps.mutableContent = true`, AND a `body_full` data field carrying the un-truncated message. The `mutable-content: 1` flag is what the Phase 3 iOS NSE consumes to optionally rewrite the body based on the on-device preview-toggle ([`docs/04-Architecture.md:535–540`](../../../docs/04-Architecture.md)). Both payloads share the same `notifications.body_data` source-of-truth — only the wire format differs.
- **NEW** End-to-end on-send-prune contract: integration test simulates an `UNREGISTERED` response from FCM and verifies the corresponding `(user_id, platform, token)` row is gone via `deleteTokenIfStale(...)` when the race-guard predicate matches. A second test exercises the race: an `UNREGISTERED` response after a re-registration that bumped `last_seen_at` past `dispatchStartedAt` MUST NOT delete the just-re-registered row. This is the contract `fcm-token-registration` deferred to "the FCM push dispatch change" — this change owns it.
- **MODIFIED** [`in-app-notifications`](../../specs/in-app-notifications/spec.md) capability — replace the requirement that says "V10 SHALL ship one implementation (`InAppOnlyDispatcher`) that is a no-op" with a successor that points at `fcm-push-dispatch` as the production implementation, while preserving the `NotificationDispatcher` seam contract verbatim (post-DB-commit dispatch, no modification of `NotificationService` or any emitter). Add a requirement that the dispatcher composition (FCM + in-app log) is wired in at DI bootstrap; choice between Remote-Config-toggled OR composite is locked in `design.md`. Tighten "in sequence" wording to "call-order, not completion-order" per `design.md` D10. Add composite catch-and-log policy per `design.md` D14.
- **MODIFIED** [`fcm-token-registration`](../../specs/fcm-token-registration/spec.md) capability — the spec currently documents the on-send 404/410 → row-delete contract as a forward-reference ("the deferred Phase 2 on-send-failure delete"). Update the cross-reference so the immediate-prune contract is now owned by `fcm-push-dispatch`, AND update the documented contract shape to reflect the round-1 design changes: prune triggers are `UNREGISTERED` + `SENDER_ID_MISMATCH` only (NOT `INVALID_ARGUMENT` per D6), and the DELETE includes the `last_seen_at <= :dispatch_started_at` race-guard predicate per D12. The weekly stale-token cleanup remains deferred to Phase 3.5.

## Out of Scope

- **Per-conversation push batching** (max 1 push per 10s per conversation; burst-merge "3 pesan baru dari {username}" copy per [`docs/04-Architecture.md § Per-Conversation Batching`](../../../docs/04-Architecture.md)). Defer because 1:1 chat hasn't shipped — there are no `chat_message` notifications to batch yet. Tracked in `FOLLOW_UPS.md`.
- **iOS NSE App Group preview-toggle** wiring on the iOS client side. The backend ships `body_full` per spec; the NSE that consumes it is Phase 3 mobile work ([`docs/04-Architecture.md § iOS NSE setup checklist`](../../../docs/04-Architecture.md)).
- **Notification-type-specific copy localization beyond a minimal first cut.** Ship Indonesian copy for the four V10-wired types (`post_liked`, `post_replied`, `followed`, `post_auto_hidden`). Other types in the [`docs/05-Implementation.md:826`](../../../docs/05-Implementation.md) enum (`chat_message`, `subscription_*`, `account_action_applied`, `data_export_ready`, `chat_message_redacted`, `privacy_flip_warning`, `username_release_scheduled`, `apple_relay_email_changed`) get a TBD/fallback copy ("Pemberitahuan baru" / "New notification") and an explicit follow-up — their emit-sites haven't shipped yet.
- **Stale token cleanup worker** (weekly `last_seen_at < NOW() - INTERVAL '30 days'` purge) — Phase 3.5 admin worker territory per [`docs/08-Roadmap-Risk.md` Phase 3.5 item 10](../../../docs/08-Roadmap-Risk.md). The on-send-prune contract (`UNREGISTERED` + `SENDER_ID_MISMATCH` per `design.md` D6, race-guarded by `last_seen_at <= :dispatch_started_at` per D12) is the only token-removal path this change owns.
- **Cross-region FCM dedup / idempotency.** No Redis-side dedupe of recently-dispatched notification IDs. The `notifications` table has UUID PK so emit-site duplication is already impossible at the source; the FCM-side multi-device fanout is intentional, not a duplication.

## Capabilities

### New Capabilities
- `fcm-push-dispatch`: server-side FCM push delivery — `:infra:fcm` module, `FcmDispatcher` impl of `NotificationDispatcher`, per-platform payload builders (Android data-only / iOS alert+mutable-content with APNs 4 KB clamp on UTF-8 codepoint boundary), shadow-ban-aware actor-username lookup via `visible_users`, and the on-send `UNREGISTERED` / `SENDER_ID_MISMATCH` → `user_fcm_tokens` row-delete contract guarded by `last_seen_at <= :dispatch_started_at` (DB-clock domain) to preserve the re-registration race. Composite catch-and-log at ERROR severity with counter metric for rate-based alerting.

### Modified Capabilities
- `in-app-notifications`: the `NotificationDispatcher` seam now has a real production implementation (`FcmDispatcher`) instead of a no-op; spec text describing "V10 SHALL ship one implementation (`InAppOnlyDispatcher`) that is a no-op" is replaced with the successor wording that admits both impls, pins composition to DI bootstrap, tightens "in sequence" to "call-order, not completion-order," and requires composite catch-and-log per `design.md` D14.
- `fcm-token-registration`: the cross-reference describing where the on-send-prune contract lives is updated from "deferred to the FCM push dispatch change" to "owned by `fcm-push-dispatch` capability." The contract shape is also updated to match the round-1 design: prune triggers are `UNREGISTERED` + `SENDER_ID_MISMATCH` only (NOT `INVALID_ARGUMENT`), and the DELETE includes the race-guard predicate.

## Impact

- **Affected modules**: new `:infra:fcm`; `:backend:ktor` Koin DI graph (wire `FcmDispatcher` in non-test environments, keep `InAppOnlyDispatcher` available for tests); `gradle/libs.versions.toml` (pin `firebase-admin` SDK + record in [`docs/09-Versions.md`](../../../docs/09-Versions.md) Version Decisions table); `settings.gradle.kts` + [`dev/module-descriptions.txt`](../../../dev/module-descriptions.txt) + `dev/scripts/sync-readme.sh --write` (new module entry per `CLAUDE.md` "Root README module list is auto-generated" rule).
- **Affected APIs**: none externally — no new endpoints, no schema changes. The `notifications` table and `user_fcm_tokens` table are sufficient as-is.
- **Affected secrets**: introduces a new secret-name convention reading via `secretKey(env, "firebase-admin-sa")`. Slot reserved in GCP Secret Manager per [`docs/08-Roadmap-Risk.md:51`](../../../docs/08-Roadmap-Risk.md); operator must populate prod + staging slots before deploy. The pre-Phase-1 task list already calls out this slot.
- **Affected lint rules**: `secretKey(env, name)` helper rule must be honored for the Firebase service-account read (already enforced; this change adds a new call-site, not a new rule). No new Detekt rules introduced.
- **Affected tests**: new module `:infra:fcm` test surface (mocked FCM Admin SDK using its built-in fakes or a stub); new integration test coverage in `:backend:ktor` exercising the dispatch-after-commit ordering and the 404/410 → row-delete contract end-to-end.
- **Backwards compatibility**: zero. Existing API surface unchanged; existing `notifications` writes continue to work; the only behavior change is "pushes now actually fire" for clients that have already registered an FCM token.
