## Why

Auth is the gate every other feature stands behind. Without JWTs, refresh rotation, and instant revocation, no user-facing endpoint can be built (every route needs `sub`). This change lands the **auth backbone as documented in `docs/05-Implementation.md` § Authentication / Session Management**: RS256 access tokens with kid rotation slots, refresh tokens with reuse detection, `users.token_version` for instant global revocation, Google/Apple ID-token verification, and the `users` + `refresh_tokens` schemas. It also bootstraps the local-dev DB stack (Docker Postgres+PostGIS) so this and every future DB-touching change has a place to run. Mobile UI, attestation, age gate, and analytics consent are explicitly deferred.

## What Changes

### Backend
- New module `:infra:supabase` (interface + DTOs only this change) holds the `UserRepository` + `RefreshTokenRepository` contract; concrete implementation uses HikariCP + plain Postgres JDBC against the local DB.
- New module `:backend:ktor` additions:
  - **JWT issuer (RS256)** with `kid` header, claims `{ sub, exp, iat, token_version, kid }`, 15-minute TTL. Private key resolved via `secretKey(env, "ktor-rsa-private-key")`; for local dev, an env-var-supplied PEM (auto-generated by a helper script). Public key exposed at `GET /.well-known/jwks.json`.
  - **Token-version check middleware**: every authenticated route compares JWT `token_version` against `users.token_version`. Stale → 401. **No Redis cache yet** (DB hit per request is fine at MVP volume; cache lands when Redis is wired for rate limiting).
  - **Sign-in endpoint** `POST /api/v1/auth/signin`: body `{ provider: "google"|"apple", id_token: string, device_fingerprint_hash?: string }`. Verifies the ID token against Google or Apple JWKS, hashes the provider subject, looks up `users` by `google_id_hash` / `apple_id_hash`, and either:
    - User exists, not banned: issue access + refresh, return `{ access_token, refresh_token, expires_in }`.
    - User exists, banned: 403 with code `account_banned`.
    - User does not exist: **404 with code `user_not_found`**. Signup creation is deferred (see Flags).
  - **Refresh endpoint** `POST /api/v1/auth/refresh`: body `{ refresh_token: string }`. Implements the rotation logic from `docs/05-Implementation.md`: 30-second overlap window, reuse detection → revoke entire `family_id` + increment `users.token_version`. Returns new access + refresh.
  - **Logout endpoint** `POST /api/v1/auth/logout`: revokes the calling refresh token (single device; family preserved).
  - **Logout-all endpoint** `POST /api/v1/auth/logout-all`: deletes all refresh tokens for the user + increments `users.token_version`.
  - **Realtime token endpoint** `GET /api/v1/realtime/token`: requires a valid Ktor RS256 JWT, then issues a Supabase-compatible HS256 JWT signed with `secretKey(env, "supabase-jwt-secret")`. TTL 1 hour, claims `{ sub, role: "authenticated", exp, iat }`.
  - **Apple S2S endpoint stub** `POST /internal/apple/s2s-notifications`: signature verified against Apple JWKS, `aud` = bundle ID, dedup via `transaction_id`. **Only handles `email-enabled` / `email-disabled` events**; `consent-revoked` and `account-delete` return 501 with a TODO log line (deferred to the deletion-flows change).
- `:backend:ktor` Ktor `Authentication` plugin installed with the RS256 verifier; `module()` wires sign-in + refresh + logout + JWKS routes.
- `secretKey(env, name)` helper extended to actually **resolve** secrets in this order: GCP Secret Manager (later changes), `<env>_<NAME_UPPERCASE_UNDERSCORED>` env var, error. For dev: env vars only.
- `/health/ready` upgraded: real probe that runs `SELECT 1` against the Postgres pool with a 500 ms timeout (per `docs/04-Architecture.md` health spec).

### Database (Flyway V2)
- `users` table per the canonical schema in `docs/05-Implementation.md` § Users Schema, **including** the `CHECK (date_of_birth <= CURRENT_DATE - INTERVAL '18 years')` defense.
- `refresh_tokens` table per § Session Management.
- All columns from the canonical schema land in V2 (no slimmed-down variant), so later features don't need ALTER. Defaults are used for columns whose feature is deferred (`analytics_consent` JSONB default, `subscription_status` default `'free'`, etc.).
- RLS policy `participants_can_subscribe` from § RLS Policy with Regex Guard is created in V2 against `realtime.messages`. **Untestable in this change** because chat doesn't exist; included so the chat change inherits the gate. The RLS unit-test suite (mandatory in `docs/05-Implementation.md`) is also deferred to the chat change.

### Local dev environment
- New `dev/docker-compose.yml` with two services:
  - `postgres` (image `postgis/postgis:16-3.4`, exposes 5432, password from `dev/.env`)
  - `redis` (image `redis:7-alpine`, exposes 6379) — included now because adding it later means another change touching infra; idle cost is zero.
- New `dev/README.md` — start/stop commands, env-var template (`dev/.env.example` committed; `dev/.env` gitignored).
- New `dev/scripts/generate-rsa-keypair.sh` — one-shot helper that emits a PEM-encoded RSA-2048 key pair + a base64-encoded `KTOR_RSA_PRIVATE_KEY` env var line for `dev/.env`.
- Architecture doc says "Supabase CLI for local parity"; this change ships **plain Postgres+PostGIS** instead (see Flags / design.md Decision 1).

### Tests
- Kotest specs:
  - `JwtIssuerTest`: round-trip sign+verify, `token_version` claim present, `kid` header set, expired token rejected.
  - `RefreshRotationTest`: happy path, 30-second overlap accepted, reuse-after-overlap revokes family + bumps `token_version`.
  - `SignInFlowTest`: existing user → 200 + tokens; unknown user → 404; banned → 403. Google/Apple JWKS calls are stubbed.
  - `HealthReadyTest`: returns 200 when Postgres reachable; returns 503 when pool times out.

## Capabilities

### New Capabilities
- `auth-jwt`: Defines the RS256 access-token format, JWKS publication, `kid` rotation slots, and the `token_version`-based instant-revocation contract.
- `auth-session`: Defines refresh-token rotation, reuse detection, family revocation, and the four logout/revocation latency targets.
- `auth-signin`: Defines the Google/Apple ID-token verification flow, the sign-in / refresh / logout HTTP contract, and the user-lookup behavior (existing-only in this change).
- `auth-realtime`: Defines the Supabase-compatible HS256 token issuance contract for chat WSS authentication.
- `users-schema`: Defines the canonical `users` table and `refresh_tokens` table introduced by Flyway V2.
- `local-dev-environment`: Defines the Dockerized Postgres+PostGIS+Redis stack, the dev secret/env-var conventions, and the keypair-bootstrap script.

### Modified Capabilities
- `backend-bootstrap`: `/health/ready` upgrades from a 200 stub to a real Postgres probe with a 500 ms timeout. Auth plugin install added to `Application.module()`.
- `migration-pipeline`: First real migration `V2__auth_foundation.sql` lands, exercising the pipeline beyond the placeholder.

## Impact

- **Code paths**: new `:infra:supabase` module, several new packages under `backend/ktor/src/main/kotlin/id/nearyou/app/` (`auth/`, `infra/db/`, `infra/jwks/`); modified `Application.kt`; new `dev/` directory; new Flyway migration; updated `application.conf` (DB connection from env).
- **Local workflow**: developers must run `dev/docker-compose.yml` before `./gradlew :backend:ktor:run`. Documented in `dev/README.md`.
- **External**: no production deployment change (no Cloud Run touched). The change leaves a working local backend that signs in seeded users.
- **Onward dependencies satisfied**: any future change can now write `routing { authenticate { ... } }` and trust `call.principal<UserPrincipal>()`. DB-touching changes have V2 schema as their starting point.

## Flags — Load-Bearing in Docs, Cut in This Proposal

I cut these to keep scope tight. **Each one is a real choice, please review.**

1. **Signup creation deferred (recommended)**. The canonical `users` insert needs `date_of_birth` (NOT NULL + 18+ CHECK), `username` (NOT NULL UNIQUE), and `invite_code_prefix` (NOT NULL UNIQUE). Each is bound to a deferred feature (age gate / username generation / referrals). Cleanest cut: **no signup endpoint yet**; only existing seeded users can sign in. Alternative: stub each (placeholder DOB `1990-01-01`, `user_<8hex>` username, HMAC-derived invite prefix) — but stubs leak across feature boundaries.
2. ~~Realtime token endpoint deferred.~~ **Pulled in** per user review. Endpoint listed above; `auth-realtime` capability added.
3. **`/internal/cleanup` job deferred**. Daily/weekly refresh-token sweeper (Cloud Scheduler → Cloud Run Jobs). Without it, `refresh_tokens` grows unbounded. At MVP volume (no users yet) it doesn't matter; lands with the deploy-pipeline change. Mark as known debt.
4. **Apple S2S deletion handlers deferred** (per your defer list). Endpoint exists but `consent-revoked` / `account-delete` return 501. Apple Developer Console submission requires the endpoint to respond 200 to all event types — **so Apple registration cannot complete until those handlers land**. Not a blocker in dev/staging.
5. **Redis token_version cache deferred**. Docs prescribe a 5-minute Redis cache with explicit invalidation. Without it, every authenticated request does one extra `SELECT token_version FROM users WHERE id = ?`. Acceptable at MVP volume; cache lands when Redis is otherwise needed (rate limiting). The Redis container is in dev compose but unused.
6. **RLS test suite deferred**. The 8 mandatory test cases in `docs/05-Implementation.md` § RLS Policy with Regex Guard target chat behavior; written in the chat change.
7. **Local dev uses plain Postgres+PostGIS, not Supabase CLI**. Architecture doc § Deployment > dev says Supabase CLI for "local parity". But our auth bypasses Supabase Auth (Ktor verifies Google/Apple directly), and Realtime/PostgREST aren't used by the backend either. Plain PG+PostGIS is faster to spin up, smaller image, and matches what Ktor actually talks to. The Supabase CLI parity gap (GoTrue, Realtime, PostgREST) is irrelevant until chat lands; we revisit then. **Recommendation**: confirm this deviation before I implement.
8. **Anomaly detection metrics deferred**. Sentry/Slack alerting on `subscribe rate per sub`, `JWT verify fail rate`, etc. Lands with the observability change.
9. **JWT key rotation procedure documented but not exercised**. The JWKS supports multiple kids; only one kid is configured initially. Rotation script lands when actually needed (post-launch operational task).
