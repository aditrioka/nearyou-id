# content-moderation-perspective-api

## Why

The text moderation pipeline at [`docs/06-Security-Privacy.md:151-180`](../../../docs/06-Security-Privacy.md) prescribes three tiers: Layer 1 (profanity blocklist, sync REJECT) and Layer 2 (UU ITE wordlist, sync soft-flag) shipped in [`content-moderation-keyword-lists` (PR #70)](../archive/2026-05-07-content-moderation-keyword-lists/proposal.md); **Layer 3 (Google Perspective API toxicity classifier, async post-INSERT, 500 ms timeout, fail-open)** is the unbuilt closer — explicitly carved out of the keyword-lists change as "this change provides the boundary that Perspective API will consume, but doesn't implement the async layer" ([`design.md:32`](../archive/2026-05-07-content-moderation-keyword-lists/design.md)). The supporting scaffolding is reserved but inert: [`docs/05-Implementation.md:523`](../../../docs/05-Implementation.md) reserves the `moderation_queue.trigger = 'perspective_api_high_score'` enum value at V9; [`docs/05-Implementation.md:1042`](../../../docs/05-Implementation.md) reserves the `perspective_api_enabled` Firebase Remote Config kill-switch flag (default TRUE per Pre-Phase 1 §18); the Pre-Launch security-review checklist at [`docs/08-Roadmap-Risk.md`](../../../docs/08-Roadmap-Risk.md) Phase 2 §16 names this change as the closer for Phase 2 moderation work. This change builds the async Layer 3 dispatch + emit-site wiring + spec contract so every persisted post / reply runs through a toxicity classifier with explicit fail-open semantics — closing the Pre-Launch security-review gap for the toxicity-classifier tier and unblocking the in-house ID-language model Month 6+ scope.

## What Changes

- **NEW** `:infra:perspective-api` Gradle module (peer to `:infra:fcm`, `:infra:remote-config`) owning the Google Perspective API HTTP client. Module exposes a single `PerspectiveApiClient` interface in pure-Kotlin types (no vendor SDK leakage) with two methods: `analyze(content: String): PerspectiveScore?` (returns `null` on timeout / API error, the fail-open signal) and `isHealthy(): Boolean` (for health-check probe wiring, optional). Implementation uses `ktor-client` HTTP POST to `https://commentanalyzer.googleapis.com/v1alpha1/comments:analyze` with the API key in the `key` query param per [Perspective API docs](https://developers.perspectiveapi.com/s/about-the-api-methods?language=en_US). Request body: `{"comment":{"text":"<content>"},"requestedAttributes":{"TOXICITY":{},"SEVERE_TOXICITY":{},"IDENTITY_ATTACK":{},"THREAT":{}},"languages":["id","en"]}`. Response: `{"attributeScores":{"TOXICITY":{"summaryScore":{"value":0.42}}, ...}}` — the client takes `max()` over the 4 requested attribute summary scores and returns it as a single `Float` in `[0.0, 1.0]`.
- **NEW** `PerspectiveDispatcher` orchestrator in `:backend:ktor` (under `id.nearyou.app.moderation`). Owns the **async post-INSERT dispatch lifecycle**: a single `CoroutineScope` with a structured-concurrency `SupervisorJob` (mirrors `FcmDispatcherScope` from `fcm-push-dispatch`), a 500 ms `withTimeoutOrNull` budget per dispatch, and the canonical fail-open path (timeout / HTTP error / parse error → no auto-hide flip, no queue row, single structured WARN log with `event=perspective_api_fail_open` `outcome=<timeout|http_error|parse_error|kill_switch_off>`). On `score > 0.8` → UPDATE `posts.is_auto_hidden = TRUE` (or `post_replies.is_auto_hidden = TRUE`) AND INSERT one `moderation_queue` row with `trigger = 'perspective_api_high_score'`, `status = 'pending'`, `priority = 5`, `notes = NULL`. On `0.6 ≤ score ≤ 0.8` → INSERT the queue row only (no auto-hide flip). On `score < 0.6` → noop. The dispatcher SHALL NOT block the user-facing HTTP response: the route handler responds 201 to the user, then enqueues the dispatch via `dispatcher.dispatchAsync(targetType, targetId, content)` and returns. Failure modes never propagate to the user. Per-dispatch lifecycle drains on JVM shutdown via the same `cancelAndJoin` pattern as `FcmDispatcherScope.shutdown`.
- **NEW** `perspective_api_enabled` Firebase Remote Config kill-switch read on each dispatch (5-min Redis cache via the existing `RemoteConfigClient` from `:infra:remote-config`). Flag OFF → noop, no API call. Cached read mirrors the `ModerationListLoader` cache shape (`{scope:perspective}:{key:enabled}` Redis hash tag).
- **NEW** Two GCP Secret Manager slot reservations: `perspective-api-key` (production) and `staging-perspective-api-key` (staging). Read via the canonical `secretKey(env, "perspective-api-key")` helper per [`openspec/project.md`](../../project.md) § Critical invariants. The `:infra:perspective-api` module's `PerspectiveApiClient` factory takes the resolved API key string at construction time — same `secretKey()`-resolved-once pattern as `FcmDispatcher`'s service-account JSON. Slot creation is operations work tracked in [`docs/10-Setup-Checklist.md`](../../../docs/10-Setup-Checklist.md); the spec scenario covers the canonical resolve-call shape.
- **NEW** `ktor-client-cio` + `ktor-client-content-negotiation` + `ktor-client-serialization-kotlinx-json` artifacts on the `:infra:perspective-api` classpath (the project doesn't yet have an outbound HTTP client; this change introduces the ktor client coordinate set, pinned in `gradle/libs.versions.toml` at the same Ktor version as the server — `3.4.x` line). The CIO engine matches the project's existing JVM-only posture (no Netty); content-negotiation + kotlinx-json align with the project's serialization standard.
- **MODIFIED** `content-moderation-keyword-lists` capability spec: appends a "Layer 3 sequencing" requirement noting that Layer 3 (Perspective API) runs **post-INSERT, async**, after the sync `TextModerator.moderate(...)` call has produced `Verdict.Allow` or `Verdict.Flag`. Layer 1 (`Verdict.Reject`) short-circuits BEFORE INSERT, so Layer 3 never runs against rejected content.
- **MODIFIED** `post-creation` capability spec: appends an "async Layer 3 dispatch" requirement to `POST /api/v1/posts` — after the canonical single-INSERT transaction commits, the route handler SHALL invoke `perspectiveDispatcher.dispatchAsync(targetType = 'post', targetId = <new_post_id>, content = <post.content>)` and respond 201 to the caller without awaiting the dispatch result. The dispatch operates on the already-persisted post and writes its results in a separate transaction.
- **MODIFIED** `post-replies` capability spec: appends the same "async Layer 3 dispatch" requirement to `POST /api/v1/posts/{post_id}/replies`, with `targetType = 'reply'` and `targetId = <new_reply_id>`.
- **OUT OF SCOPE — NOT MODIFIED:** `chat-conversations`. Per V15 schema, `chat_messages` lacks an `is_auto_hidden` column — Layer 3's auto-hide path is structurally inapplicable. Layer 1 (Reject) + Layer 2 (Flag) via the existing `TextModerator.moderate(...)` continue to apply. Admin redaction (`redacted_at` / `redacted_by`) is the chat moderation surface for severe content. A future change MAY add Perspective API to chat by either (a) adding `is_auto_hidden` to `chat_messages` (schema change) or (b) routing `> 0.8` chat scores to the existing redaction path automatically. Both paths are out of scope here; documented in `design.md` Open Question Q1.

## Capabilities

### New Capabilities

- `content-moderation-perspective-api`: the `:infra:perspective-api` HTTP client, the `PerspectiveDispatcher` async orchestrator, the kill-switch read path, the score-threshold decision rules (`> 0.8` auto-hide + queue; `[0.6, 0.8]` queue only; `< 0.6` noop), the fail-open contract on every error path (timeout, HTTP error, parse error, kill-switch off), the structured WARN log shape, the structured-concurrency lifecycle including JVM-shutdown drain.

### Modified Capabilities

- `content-moderation-keyword-lists`: appends a Layer 3 sequencing requirement so the existing pipeline contract reflects all 3 tiers in canonical order (Reject pre-INSERT → INSERT → Async Layer 3 post-INSERT).
- `post-creation`: appends the async dispatch invocation requirement so the canonical write path reflects the post-INSERT Perspective hop.
- `post-replies`: appends the same async dispatch invocation requirement.

(`moderation-queue` is **NOT** modified by this change. The `'perspective_api_high_score'` trigger value is already reserved at V9 and the queue's INSERT shape is unchanged. `chat-conversations` is **NOT** modified — Layer 3 is out of scope for chat per the schema-shape constraint above.)

## Impact

**Code**:
- New Gradle module `:infra:perspective-api` (~2-3 source files: `PerspectiveApiClient` interface, `KtorPerspectiveApiClient` implementation, `PerspectiveScore` data class). Mirrors the `:infra:fcm` / `:infra:remote-config` shape — pure-Kotlin public surface, vendor SDK or HTTP client encapsulated.
- New code in `:backend:ktor` for `PerspectiveDispatcher` + Koin wiring + emit-site invocation patches in the post-creation + post-replies handlers (small ~5 LOC additions per emit site).
- New `gradle/libs.versions.toml` entries for the Perspective API key slot name + the Ktor client coordinates.
- New tests: `:infra:perspective-api` HTTP client (request shape, response parse, timeout fail-open, HTTP error fail-open, max-attribute scoring), `:backend:ktor` `PerspectiveDispatcher` (async dispatch, kill-switch off → noop, score thresholds → correct DB writes, structured-concurrency drain on shutdown, no exception propagates to caller), integration tests for both emit sites (post + reply auto-hide on hi-score, queue-only on mid-score, allow on lo-score), structural call-order assertion (dispatcher is invoked AFTER the INSERT commit and BEFORE the response).

**APIs**:
- No new REST endpoints. No new error response shapes. The `POST /api/v1/posts` and `POST /api/v1/posts/{post_id}/replies` HTTP envelopes are unchanged from the caller's perspective — the async Layer 3 dispatch is invisible to the user (post is persisted + visible to author immediately; auto-hide flip happens behind the scenes within ~500 ms if Perspective scores high). The existing `Verdict.Reject` / `Verdict.Flag` from the sync `TextModerator` continue to produce the same HTTP responses.

**Dependencies**:
- New direct dependency on `ktor-client-cio` + `ktor-client-content-negotiation` + `ktor-client-serialization-kotlinx-json` (pinned at the Ktor server version, 3.4.x line). Justification logged in [`docs/09-Versions.md`](../../../docs/09-Versions.md) per the Version Pinning Decisions Log convention.
- New Secret Manager slots: `perspective-api-key` (prod) + `staging-perspective-api-key` (staging). Slot creation is operations work; the spec covers only the canonical `secretKey(env, "perspective-api-key")` resolve call.
- Reuses existing `:infra:redis` (kill-switch cache), `:infra:remote-config` (kill-switch read), `:lint:detekt-rules` (`SecretsKeyRule` enforces the `secretKey()` helper convention), Sentry surface (structured WARN log).

**Systems**:
- `moderation_queue` writer count grows from 2 (auto-hide-3-reports + UU ITE keyword) to 3 (adds Perspective API). Admin-triage volume increase expected to be modest at MVP scale (≤500 seed users in soft launch); the > 0.8 threshold is conservative enough that false-positive volume is bounded.
- Pre-Launch security-review checklist item closes for the toxicity-classifier tier; Layer 3 covers the patterns the keyword list misses (tone shift, leet substitution, novel slurs).
- Phase 2 §16 closes; the only remaining Phase 2 product item.
- Free-tier 1 QPS Perspective API quota is well above platform write velocity at MVP scale (10/day likes × N users + 20/day replies × N users + 50/day chat × N users — with chat OUT of scope, the per-second rate is trivial). No global throttler needed for V1; revisit if MAU > 10k or per-user limits loosen.
- Follow-ups unblocked / created by this change:
  - `content-moderation-perspective-api-chat-coverage` (Q1 follow-up): if telemetry shows chat is a meaningful toxicity vector, file a change adding `is_auto_hidden` to `chat_messages` OR routing high-score chat to the existing admin redaction path.
  - `content-moderation-perspective-api-detekt-rule` (rule-of-three): once a fourth content-write site appears (e.g., post edit content, future Premium-username Perspective scoring), the existing `content-write-moderation-detekt-rule` follow-up gets extended to also enforce Layer 3 dispatch invocation post-INSERT.
  - `content-moderation-perspective-api-throttler`: if MAU > 10k OR per-user content limits loosen meaningfully, add a global Lettuce-backed token bucket at 1 QPS (free tier) or N QPS (paid tier) before invoking the API.
- Existing `perspective-api-stopgap` follow-up entry in `FOLLOW_UPS.md` gets deleted on archive (this change is the completion of that entry's action items).
