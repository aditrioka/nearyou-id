## Why

`nearby-timeline-with-blocks` (V5) shipped Nearby, `following-timeline-with-follow-cascade` (V6) shipped Following. Global is the third and final timeline tab and the only one guests can ever see in the product spec (`docs/02-Product.md` §3). Phase 1 item 15 in `docs/08-Roadmap-Risk.md` explicitly flags *"Global remains (needs `admin_regions` polygon reverse-geocoding)"* — it is blocked on one piece of schema + data (the `admin_regions` polygon table) and the reverse-geocode path that every Global list item depends on to render its city label. Shipping Global also pays down Phase 2 item 2 in one go. Authenticated-only scope keeps the change tight; guest read-only access + 10/session + 30/hour scroll caps require the Redis rate-limit infrastructure (Phase 1 item 24) and are deferred to that change.

## What Changes

- Add the next Flyway migration (first unused `V<N>__admin_regions.sql` after the latest shipped version) introducing:
  - `admin_regions` table exactly per `docs/05-Implementation.md` lines 1007–1021: `id`, `name`, `level` (`province` | `kabupaten_kota`), `parent_id` self-FK, `geom GEOGRAPHY(MULTIPOLYGON)`, `geom_centroid GEOGRAPHY(POINT)`; GIST indexes on `geom` and `geom_centroid`; btree indexes on `level` and `parent_id`.
  - A Flyway-time `admin_regions` seed derived from the chosen kabupaten/kota polygon dataset (BPS primary, OSM fallback — see `design.md` for the final pick per Open Decision #4 in `docs/08-Roadmap-Risk.md`). DKI Jakarta's 5 kotamadya + Kepulauan Seribu are inserted at `level = 'kabupaten_kota'` as specified in `docs/02-Product.md`. Coastal kabupaten polygons are extended by a 12 nautical mile (~22 km) maritime buffer at import time per `docs/02-Product.md:200–203`, so posts at sea near the shoreline still match step 1 of the fallback ladder.
  - The `posts_set_city_tg` BEFORE INSERT trigger (described below). Schema additions on `posts` are **not** needed — `posts.city_name TEXT` and `posts.city_match_type VARCHAR(16)` already exist on the table from V4 ([`backend/ktor/src/main/resources/db/migration/V4__post_creation.sql:22–23`](backend/ktor/src/main/resources/db/migration/V4__post_creation.sql)). V11 is strictly additive: new reference table + new trigger + idempotent `visible_posts` view refresh. No `ALTER TABLE posts`.
- Add `GET /api/v1/timeline/global?cursor=` — Bearer JWT required (this change), keyset cursor on `(created_at DESC, id DESC)` identical in shape to Nearby + Following, `LIMIT 31` (per-page cap 30), `FROM visible_posts` with bidirectional `user_blocks` NOT-IN subqueries so `BlockExclusionJoinRule` passes on the new query literal. No geo params, no spatial filter, no radius, no envelope validation, no `distance_m` in the response. Response item shape matches Nearby + Following minus `distance_m` plus a required `city_name` string (`""` when the author's post has no resolved region).
- Carry the V7 `LEFT JOIN post_likes` (`liked_by_viewer`) and V8 `LEFT JOIN LATERAL` reply counter (`reply_count`) with identical cardinality + shadow-ban + block-state invariants already enforced on Nearby and Following. No new variance from the shipped pattern.
- Populate `posts.city_name` + `posts.city_match_type` on **write** via a Flyway-time trigger `posts_set_city_tg` (BEFORE INSERT) implementing the **4-step fallback ladder** from [`docs/02-Product.md:192–196`](docs/02-Product.md):
  1. `ST_Contains(geom, NEW.actual_location)` strict match → `city_match_type = 'strict'`.
  2. If step 1 returns no row: `ST_DWithin(geom, NEW.actual_location, 10)` (10-meter buffer) ordered by `ST_Distance(geom_centroid, NEW.actual_location) ASC LIMIT 1` → `city_match_type = 'buffered_10m'`. Resolves floating-point boundary artifacts with a deterministic tie-breaker toward the centroid.
  3. If step 2 returns no row: `ST_DWithin(geom, NEW.actual_location, 50000)` (50km) ordered by `ST_Distance(geom, NEW.actual_location) ASC LIMIT 1` → `city_match_type = 'fuzzy_match'`. Catches coastal posts, polygon-gap regions, and off-shore posts beyond the 12nm buffer.
  4. If step 3 returns no row: both `city_name` and `city_match_type` stay NULL. Client renders `""` in the Global response; app-level UX may substitute "Indonesia" / "Luar Indonesia" per `docs/02-Product.md:196`.
  
  Trigger reads `NEW.actual_location` (sanctioned admin-side / reverse-geocoding use; not a `display_location` violation — accuracy matters at the kotamadya scale where 500m jitter crosses polygon boundaries). All four steps happen DB-side inside one trigger function; Global's hot path is an unaffected single keyset read against `visible_posts`. Legacy V4 posts are left NULL deliberately; a one-shot backfill job is called out as optional in `tasks.md` and NOT required for this change to ship.
- Extend the `CoordinateJitterRule` / `actual_location` policy documentation to explicitly list the post-creation trigger as a second sanctioned reader of `actual_location` alongside the existing admin path. No lint-rule behavior change; KDoc only.
- Update `post-creation` spec with a new requirement covering the trigger population of `city_name` + `city_match_type` on INSERT via the 4-step fallback, the NULL-tolerant behavior for legacy rows and step-4 polygon-gap outcomes, and the provenance semantics of `city_match_type`.

## Capabilities

### New Capabilities
- `global-timeline`: HTTP contract for `GET /api/v1/timeline/global` — auth, optional `cursor`, keyset pagination, canonical query requirement (`FROM visible_posts` + bidirectional `user_blocks` NOT-IN + V7 likes LEFT JOIN + V8 reply counter LEFT JOIN LATERAL), per-page cap, ordering invariant, response shape (Nearby-minus-`distance_m` plus `city_name`), integration test coverage. Guest access and Redis session/hour caps are explicitly deferred; the authenticated endpoint is the full scope.
- `region-polygons`: `admin_regions` table schema (columns, constraints, indexes, seed contract); polygon dataset source decision log + 12nm maritime buffer during import; 4-step reverse-geocoding fallback ladder (strict → buffered_10m → fuzzy_match → NULL); `posts_set_city_tg` BEFORE INSERT trigger contract that populates the pre-existing `posts.city_name` + `posts.city_match_type` columns (V4); idempotent re-seed behavior (self-FK on `parent_id` survives re-import); null-tolerance for legacy posts and step-4 polygon-gap outcomes.

### Modified Capabilities
- `post-creation`: INSERT path now populates `city_name` + `city_match_type` via `posts_set_city_tg`'s 4-step fallback. New requirement + scenarios: post inside a kabupaten/kota polygon → `city_match_type = 'strict'`; post on a polygon boundary → step 2 `'buffered_10m'`; post in a coastal gap within 50km → step 3 `'fuzzy_match'`; post beyond all polygons (open ocean, EEZ) → both fields NULL without failing the INSERT; the trigger reads `actual_location` (not `display_location`).
- `users-schema`: no change to the `posts` row shape at V11 — the `city_name TEXT` and `city_match_type VARCHAR(16)` columns already exist from V4; `admin_regions` is a new reference table not FK'd to `users`. Scenario coverage verifies V11 does NOT alter `users` or add/rename any existing `posts` column.
- `visible-posts-view`: the view already projects `posts.city_name` + `posts.city_match_type` via `SELECT *` from V4. V11 re-runs `CREATE OR REPLACE VIEW visible_posts AS SELECT * FROM posts WHERE is_auto_hidden = FALSE` as an idempotent no-op refresh so `pg_views.definition` reflects V11's audit timestamp. No filter change; no break to existing consumers.
- `migration-pipeline`: the new migration is the first one to land a spatially-seeded reference table (`admin_regions`) AND a BEFORE INSERT trigger — document the precedent (dataset licensing note carried in migration header; Flyway `COPY` / `ST_GeomFromGeoJSON` pattern).
- `block-exclusion-lint`: a new protected query literal (the canonical Global query). No rule behavior change; confirmation that both directional `user_blocks` NOT-IN subqueries appear on the new timeline SQL so lint continues to pass.
- `coordinate-jitter`: the admin-/reverse-geocode-only allowlist for `actual_location` is extended to include the `posts_set_city_tg` trigger path. Spec note clarifies that this is a DB-side read (never surfaced to clients; never projected into any user-facing response). No behavior change.

## Impact

- **Code**: new `backend/ktor/.../timeline/GlobalTimelineService.kt` + route registration in the existing timeline routes file; minor `Post` DTO extension (`city_name: String`); no change to `PostsRepository.insert` — the trigger populates `city_name` + `city_match_type` server-side without repository involvement (`city_match_type` stays internal to DB + admin module; client-facing DTOs only project `city_name`). `NearbyTimelineService` + `FollowingTimelineService` response DTOs gain `city_name` (consistent payload shape across all three timelines is easier for mobile) — these two services update their `SELECT` to project `city_name` but the Nearby/Following **response** behavior is otherwise unchanged.
- **Schema**: one new Flyway migration (`V<next>__admin_regions.sql`) with `admin_regions` table + seed (polygons with 12nm maritime buffer applied at import time) + `posts_set_city_tg` trigger implementing the 4-step fallback ladder + idempotent `visible_posts` view refresh. **No `ALTER TABLE posts`** — the target columns (`city_name`, `city_match_type`) already exist from V4. `admin_regions` rows have no FK from `posts`; a hypothetical admin-driven `admin_regions` delete does NOT cascade — posts retain their snapshot `city_name` string and their `city_match_type` tag.
- **APIs**: one new endpoint `GET /api/v1/timeline/global`. Error codes: `400 invalid_cursor` (reused from Nearby + Following), `401 unauthenticated`. Nearby + Following response JSON gains `city_name` (additive; clients that ignore unknown fields are unaffected; documented in migration/changelog).
- **Dependencies**: no new libraries. PostGIS `ST_Contains` + polygon storage is already in the stack (Nearby uses `ST_DWithin`; reverse geocode shares the engine). A one-time data source pull during migration authoring — BPS kabupaten/kota GeoJSON or OSM admin-level-5 extract.
- **Out of scope (explicit)**:
  - Guest Global access (no-login read path) — requires guest JWT issuance with pre-issuance rate limits; lands with the rate-limit / guest-auth change.
  - Redis-backed 10/session soft + 30/hour hard scroll caps — same deferral as Nearby and Following; waits on Phase 1 item 24.
  - Province-level filter or island-level filter on Global — MVP is chronological-only (`docs/02-Product.md` §3 *"No province/island filter"*).
  - Ranking or algorithmic ordering — MVP is strictly `created_at DESC` (`docs/02-Product.md` §3 *"No ranking/algorithm in the early phase"*).
  - Backfilling `city_name` + `city_match_type` on legacy pre-this-change posts — a one-shot `UPDATE posts SET ...` job running the same 4-step ladder is noted as optional in `tasks.md`; the Global endpoint tolerates NULL `city_name`.
  - Province-polygon-level matching — the trigger only queries `level = 'kabupaten_kota'` rows on all 4 steps; province rows are seeded for the `parent_id` hierarchy (useful for future drill-downs and the display-location governance audit) but not referenced by the trigger.
  - Redis `geocode:{geocell:<lat2dp>_<lng2dp>}` cache layer described in `docs/02-Product.md:205` — premature for MVP write rate; the trigger's 4-step DB work is in the 1–5ms range per INSERT. Deferred to a scale-phase change alongside the broader Redis rate-limit infrastructure.
  - Perspective API text moderation (Phase 2 item 16) — separate change.
  - FCM push for new Global posts — not in the product spec; Global is polled, not pushed.
  - Mobile UI wiring for the Global tab (cross-runtime task; lands with the mobile-app change in Phase 3).
  - Admin-side polygon editor (Phase 3.5 admin panel work).
  - Reverse-geocode at read time as a fallback — the trigger + denormalized column is the only supported path; if the trigger ever fails to populate (polygon gap), the row stays NULL forever unless the backfill job is run.
