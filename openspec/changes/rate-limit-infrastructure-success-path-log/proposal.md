## Why

The `rate-limit-infrastructure` capability spec implies that every `tryAcquire` / `tryAcquireByKey` admit emits a structured log carrying `key=<key>` (per the shipped scenario "tryAcquireByKey omits userId from telemetry" at [`openspec/specs/rate-limit-infrastructure/spec.md:81-83`](../../specs/rate-limit-infrastructure/spec.md)), but the production [`RedisRateLimiter.admit()`](../../../infra/redis/src/main/kotlin/id/nearyou/app/infra/redis/RedisRateLimiter.kt) only emits structured `key=` logs on FAILURE paths (`event=redis_acquire_failed`, `event=redis_connect_failed`). Healthy Redis = no `key=` log line, so the Cloud Logging filter shape implied by the spec — and required by the §6.5 smoke verification of the just-merged [`rate-limit-ip-hashing` change (PR #74)](https://github.com/aditrioka/nearyou-id/pull/74) — returns zero hits in steady state. This drift is operator-facing: the `key=` field is the rate-limit hot-key triage signal, and missing it on the success path forces operators to reach for unit-test source archeology (currently the only structural proof) instead of querying the runtime surface the spec advertises.

## What Changes

- **MODIFIED capability**: `rate-limit-infrastructure` — ADD two new requirements: (1) a positive log-shape requirement on the admit success branch (DEBUG-level structured log `event=rate_limit_check key=<key> outcome=<allowed|rate_limited> remaining_or_retry_after=<int>`) with **seven scenarios** covering user-axis success, key-axis success without user_id, user-axis rate-limited, key-axis rate-limited, fail-soft early-return non-emission, DEBUG-filtered non-emission, and source-level structural enforcement of the if/else conditional shape; (2) a test-coverage requirement enumerating **seven success-path test scenarios** to be added to `RedisRateLimiterTelemetryTest` plus **three discoverability/pollution-prevention/structural-extension scenarios** at the spec level. The existing "tryAcquireByKey omits userId from telemetry" scenario in the `RateLimiter interface` requirement is NOT literally modified — its WHEN clause "AND a structured log is emitted" was previously vacuously satisfied in steady state (no log emitted on success); the new ADDED requirements guarantee log emission on Redis-evaluated outcomes (user-axis + key-axis × success + rate-limited = 4 paths), making the existing scenario concrete rather than vacuous. The fail-soft early-return path (Redis unreachable → `Allowed(capacity)`) is explicitly excluded with a non-emission scenario — the title "every Redis-evaluated outcome" scopes the requirement away from that path, where the existing `event=redis_connect_failed` WARN log is the operator signal.
- **Implementation**: extend [`RedisRateLimiter.admit()`](../../../infra/redis/src/main/kotlin/id/nearyou/app/infra/redis/RedisRateLimiter.kt) to emit the new `event=rate_limit_check` DEBUG log on both success branches (allowed + rate_limited), conditional on `telemetryUserId != null` for the `user_id={}` field — mirrors the existing failure-path conditional discipline at lines 145-160 byte-for-byte.
- **Tests**: extend [`RedisRateLimiterTelemetryTest`](../../../infra/redis/src/test/kotlin/id/nearyou/app/infra/redis/RedisRateLimiterTelemetryTest.kt) with success-path scenarios using a connected real-Redis client + DEBUG logger appender (mirroring the existing `RedisRateLimiterIntegrationTest` precedent for real-Redis-backed assertions). Add a source-level structural check that catches future refactors that drop the conditional and would silently re-leak `user_id=` on key-axis calls.
- **Smoke re-verification (pre-archive)**: re-run the `rate-limit-ip-hashing` §6.5 staging smoke against the post-amendment deploy — 5 sequential `GET /health/live` requests, then Cloud Logging filter `event=rate_limit_check AND key:"{ip:"` over the smoke window. Confirm 5 hits with hashed-IP form, no `user_id=`, no raw IPv4 dotted-quad. Closes the §6.5 verification gap.

No new capability is introduced. No schema, migration, or REST API surface changes. The `releaseMostRecent` path is intentionally out of scope (it is not a rate-limit-decision call site, and no operator filter shape requires symmetric coverage; future call-site needs ship in a separate change).
