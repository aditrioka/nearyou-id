## Why

Phase 2 §13 in [`docs/08-Roadmap-Risk.md:167`](../../../docs/08-Roadmap-Risk.md) specifies "OpenTelemetry instrumentation (Ktor server, HTTP client, Postgres JDBC, Redis Lettuce, Resend HTTP)" as the next step after the chat track closes (chat-foundation → chat-rate-limit → chat-realtime-broadcast → chat-message-notification, all shipped as of PR [#65](https://github.com/aditrioka/nearyou-id/pull/65)). Until OTel ships, every shipped capability that needs failure-mode visibility falls back to "structured WARN logs are the canonical observability surface" — explicitly called out at [`openspec/specs/chat-realtime-broadcast/spec.md:228`](../../specs/chat-realtime-broadcast/spec.md): _"OTel-based observability is deferred to a future `observability-otel-foundation` change."_ The same WARN-log-only carve-out implicitly applies to `fcm-push-dispatch` (5 WARN scenarios per [`openspec/specs/fcm-push-dispatch/spec.md`](../../specs/fcm-push-dispatch/spec.md)) and the internal-endpoint-auth WARN logs at [`openspec/specs/internal-endpoint-auth/spec.md:18,101`](../../specs/internal-endpoint-auth/spec.md).

Beyond resolving the existing forward reference, the change unblocks Phase 2 §14 ("Phase 2 benchmark expanded — target p95 <200ms timeline, p99 <3s cold start, auth check <30ms, spatial query isolated <50ms"). Per-span latency attribution is a hard prerequisite — the benchmark cannot identify a regressing layer without traces. The full instrumentation contract is canonical at [`docs/04-Architecture.md:386-399`](../../../docs/04-Architecture.md) (mandatory spans, mandatory attributes, propagation), and the sampling profile is canonical at [`docs/05-Implementation.md:2042`](../../../docs/05-Implementation.md) ("100% head sampling in dev Phase 2, 10% base + 100% error/slow in production").

## What Changes

- **NEW `:infra:otel` module** scaffolded under `infra/otel/`. The module is named in [`openspec/project.md:53`](../../project.md) and [`docs/04-Architecture.md:118`](../../../docs/04-Architecture.md) but does not yet exist on disk (`infra/` currently has only `fcm`, `oidc`, `redis`, `supabase`). Module exposes a single `OtelBootstrap` startup entrypoint plus minimal interceptor wiring; vendor-specific code (OTLP exporter, Grafana Cloud auth) lives entirely inside `:infra:otel` so a vendor swap (Open Decision #12) is a within-module change.
- **OTel SDK initialization at Ktor startup** with auto-instrumentation for Ktor HTTP server, HTTP client (the JDK/Apache client used by Resend + Supabase REST + Supabase Realtime publish), Postgres JDBC via HikariCP, Redis Lettuce, and the Firebase Admin SDK FCM send call. Initialization is idempotent and exits cleanly when the OTel exporter token is absent (local dev — see Sampling/Exporter section below).
- **Mandatory spans** per [`docs/04-Architecture.md:397`](../../../docs/04-Architecture.md): HTTP request on Ktor (root span), Supabase API calls (REST + Realtime broadcast publish), Redis calls (rate-limit Lua scripts + cache reads), PostGIS spatial queries (timeline reads), FCM Admin SDK send. Auto-instrumentation covers the bulk; the few manual spans needed (Realtime publish, rate-limit Lua call) are wrapped via the new `:infra:otel` `withSpan(name, attrs) { ... }` helper. Resend HTTP and CF Images calls are listed in the canonical doc but their producing modules (`:infra:resend`, `:infra:cloudflare-images`) have not shipped yet (Resend is unscaffolded; CF Images is gated by `image_upload_enabled=FALSE` per Phase 4 §16). When those capabilities ship, their changes ADOPT the cross-cutting OTel surface from this change — they don't re-introduce it. Out-of-scope here.
- **Mandatory attributes** per [`docs/04-Architecture.md:398`](../../../docs/04-Architecture.md): `user.id` (SHA-256 hashed at the span-attribute setter — never the raw UUID — using a deterministic project-wide `hashUserId(uuid)` helper); `endpoint` (Ktor route pattern, NOT raw URL — query strings + path params stripped); `db.statement` (parameterized only — raw values stripped via the JDBC instrumentation's `statementSanitizer` setting); `supabase.realtime.channel`; `geo.cloud_region` (Cloud Run region from the `K_SERVICE_REGION` env var). Standard W3C semconv attributes (`http.method`, `http.route`, `http.status_code`, `db.system`, `messaging.operation`) come from auto-instrumentation.
- **Forbidden attributes** (defense-in-depth — captured as a positive requirement so the spec encodes the negative space): NEVER attach raw `user_id` UUIDs, raw client IPs (`CF-Connecting-IP` value), raw JWT/refresh tokens, raw bearer values, JWKS contents, raw post content, raw chat message content, raw search queries, or raw Supabase admin keys to spans. The `WARN log token correlation id` pattern from [`internal-endpoint-auth/spec.md:18`](../../specs/internal-endpoint-auth/spec.md) (truncated SHA-256) is the canonical anonymization shape.
- **W3C Trace Context propagation** on all outbound HTTP from `:backend:ktor` (Resend, Supabase REST, Supabase Realtime publish, FCM Admin SDK). Auto-instrumentation handles this for the JDK/Apache HTTP client; the FCM Admin SDK uses its own internal HTTP client and gets a manual `traceparent` header injection via the `withSpan` wrapper.
- **Sampling profile** per [`docs/05-Implementation.md:2042`](../../../docs/05-Implementation.md):
  - `dev`: head 100% (full traces, OTLP exporter target = console / no-op when token absent).
  - `staging`: head 100% (Phase 2 benchmark surface).
  - `production`: 10% base ratio (`ParentBased(TraceIdRatioBased(0.1))`) + force-keep on error (`http.status_code >= 500` or `exception.escaped = true`) + force-keep on slow (root-span `duration > 500ms`). Implemented via a custom `Sampler` composed of `ParentBased(ratio)` for the base path plus a `RecordOnly + post-export decision` shape (the OTel SDK's `RecordOnly` sampler combined with a `SpanProcessor` that promotes RecordOnly spans to exported when error/slow predicates hit).
- **Exporter wiring**: OTLP/HTTP to Grafana Cloud Tempo as the recommended default per Open Decision #12 ([`docs/08-Roadmap-Risk.md:586`](../../../docs/08-Roadmap-Risk.md)). Endpoint + token read via the existing `secretKey(env, name)` helper from two new GCP Secret Manager slots: `otel-grafana-otlp-token` (production) and `staging-otel-grafana-otlp-token` (staging). When the token is absent (local dev or unconfigured staging), the exporter falls back to a no-op (logs a single startup INFO line and continues).
- **Replaces "WARN-only" carve-outs** in two shipped specs (delta files in this change):
  - `chat-realtime-broadcast`: replace the explicit deferral at line 228 with a positive requirement — on `chat_realtime_publish_failed` WARN log emission, an OTel span SHALL also record an event with the same `event` attribute and `error.type = error_class`. The crash-window contract (publish dies before WARN line fires) remains unchanged: no span on a process crash either.
  - `fcm-push-dispatch`: each of the 5 WARN scenarios (INVALID_ARGUMENT, transient errors, unknown errors, partial-failure per-token, dispatch-after-shutdown) SHALL also record a span event with the matching `event` field. Pair is best-effort — a span recording failure SHALL NOT block dispatch nor surface as an exception.
- **Version Pinning Decisions Log entries** in [`docs/09-Versions.md`](../../../docs/09-Versions.md) for each new dependency: `opentelemetry-sdk`, `opentelemetry-exporter-otlp`, `opentelemetry-instrumentation-ktor` (or manual semconv shim if Ktor 3.4 instrumentation isn't published), `opentelemetry-instrumentation-jdbc`, `opentelemetry-instrumentation-lettuce-5_1`, plus the supporting `opentelemetry-api` / `opentelemetry-context` BOM-managed pins. Pinning rationale follows the existing log shape (provenance, version-as-of-pin, rationale, next review).

## Capabilities

### New Capabilities

- `observability-otel-foundation`: OpenTelemetry tracing foundation for `:backend:ktor` — initialization contract, mandatory spans, mandatory + forbidden attributes, W3C propagation, sampling profile per environment, OTLP exporter wiring, fallback-to-no-op when exporter unconfigured. The cross-cutting source of truth referenced by per-capability span requirements; future per-capability changes consume this rather than restating sampling/exporter mechanics.

### Modified Capabilities

- `chat-realtime-broadcast`: drop the line-228 explicit deferral ("SHALL NOT emit OpenTelemetry spans in this change"); replace with a positive pairing requirement — the publish call records a span; on publish failure, the span records an event with the same `event = "chat_realtime_publish_failed"` attribute and `error.type` matching the WARN log's `error_class`. The handler-completion-required preservation rule from line 232–242 (no WARN log on crash, no span on crash either) is preserved.
- `fcm-push-dispatch`: each of the 5 WARN-log scenarios (INVALID_ARGUMENT, retryable error, unknown error, per-token partial-failure, dispatch-after-shutdown) SHALL also record a span event with the matching `event` attribute. Pair is best-effort — span-recording failure SHALL NOT propagate to the caller, SHALL NOT block dispatch, and SHALL be silently swallowed (consistent with the existing FATAL→ERROR+metric downgrade for the composite dispatcher per [`fcm-push-dispatch/spec.md:425`](../../specs/fcm-push-dispatch/spec.md)).

## Impact

- **Code**: new `:infra:otel` module (~5 source files: `OtelBootstrap.kt`, `OtelConfig.kt`, `SamplingProfile.kt`, `WithSpan.kt`, `UserIdHasher.kt`); Ktor startup wiring in `Application.module()`; one Koin DI binding for the SDK; HikariCP `MetricRegistry` swap to OTel; Lettuce `ClientResources` wired with the OTel tracer; manual `withSpan` wrapping of the Realtime publish call (replacing the existing inline call) and the rate-limit Lua call.
- **APIs**: zero contract change. Instrumentation is observable side-effect only.
- **Schema**: zero migrations.
- **Dependencies**: new pins in [`gradle/libs.versions.toml`](../../../gradle/libs.versions.toml) for the OTel SDK + exporter + instrumentation BOM. Each pin carries a row in [`docs/09-Versions.md`](../../../docs/09-Versions.md) Version Pinning Decisions Log per the project policy.
- **Mobile**: zero change. KMP-side OTel is explicitly post-MVP (see Non-goals in `design.md`).
- **Secrets**: 2 new GCP Secret Manager slots (`otel-grafana-otlp-token`, `staging-otel-grafana-otlp-token`). The setup checklist at [`docs/10-Setup-Checklist.md:220-229`](../../../docs/10-Setup-Checklist.md) already reserves these.
- **Cost**: per the projection at [`docs/01-Business.md:381`](../../../docs/01-Business.md), Grafana Cloud at Phase 2 traffic is 0 IDR (free tier sufficient); 0.2% of revenue at scale per [`docs/01-Business.md:455`](../../../docs/01-Business.md).
- **Out-of-scope** (Non-goals — captured in design.md):
  - Mobile-side OTel (KMP). Sentry KMP already covers crash + error reporting.
  - OTel Logs SDK. Structured JSON logging stays as-is; this change covers traces only.
  - Tail sampling via OTel Collector. Architecture mentions "Tail sampling via OTel Collector if volume is high" as a *future* capacity step ([`docs/04-Architecture.md:394`](../../../docs/04-Architecture.md)).
  - Metrics SDK. GCP Cloud Monitoring + Supabase dashboards already cover metrics.
  - Vendor-final decision (Grafana Cloud vs Honeycomb vs Cloud Trace, Open Decision #12). This change ships Grafana Cloud as the recommended default; vendor swap is a future within-module change in `:infra:otel`.
  - Phase 2 §14 benchmark itself. This change unblocks it but doesn't run it.
