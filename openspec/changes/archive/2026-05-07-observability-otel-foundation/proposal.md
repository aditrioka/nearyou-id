## Why

Phase 2 §13 in [`docs/08-Roadmap-Risk.md:167`](../../../docs/08-Roadmap-Risk.md) specifies "OpenTelemetry instrumentation (Ktor server, HTTP client, Postgres JDBC, Redis Lettuce, Resend HTTP)" as the next step after the chat track closes (chat-foundation → chat-rate-limit → chat-realtime-broadcast → chat-message-notification, all shipped as of PR [#65](https://github.com/aditrioka/nearyou-id/pull/65)). Until OTel ships, every shipped capability that needs failure-mode visibility falls back to "structured WARN logs are the canonical observability surface" — explicitly called out at [`openspec/specs/chat-realtime-broadcast/spec.md:228`](../../specs/chat-realtime-broadcast/spec.md): _"OTel-based observability is deferred to a future `observability-otel-foundation` change."_

Beyond resolving the existing forward reference, this change unblocks Phase 2 §14 ("Phase 2 benchmark expanded — target p95 <200ms timeline, p99 <3s cold start, auth check <30ms, spatial query isolated <50ms") at staging head-100% sampling. Per-span latency attribution is a hard prerequisite — the benchmark cannot identify a regressing layer without traces.

The full instrumentation contract is canonical at [`docs/04-Architecture.md:386-399`](../../../docs/04-Architecture.md). The sampling profile target is canonical at [`docs/05-Implementation.md:2042`](../../../docs/05-Implementation.md) ("100% head sampling in dev Phase 2, 10% base + 100% error/slow in production"). Because the project's MVP infrastructure does not include an OTel Collector (the architecture doc defers it at [`docs/04-Architecture.md:394`](../../../docs/04-Architecture.md): _"Tail sampling via OTel Collector if volume is high"_), and because correctly preserving trace_id linkage on force-keep promotion requires Collector tail sampling, **this change ships the foundation only with a simple `ParentBased(TraceIdRatioBased(0.1))` production sampler — no force-keep promotion**. The "100% errors + 100% slow" tail of the canonical sampling profile is **deferred to a focused follow-up change** that deploys the Collector and reworks promotion correctly. Phase 2 §14 benchmarking proceeds against staging at head-100% sampling, where the force-keep tail is unnecessary.

This rescope is the result of three review rounds; round 3 surfaced four hard blockers (codebase-grounded findings) that each warranted a focused proposal rather than inline patches. The follow-up changes are tracked in [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md).

## What Changes

- **NEW `:infra:otel` module** scaffolded under `infra/otel/` (package `id.nearyou.app.infra.otel` per the existing `:infra:fcm` / `:infra:redis` / `:infra:supabase` convention). The module is named in [`openspec/project.md:53`](../../project.md) and [`docs/04-Architecture.md:118`](../../../docs/04-Architecture.md) but does not yet exist on disk. Module exposes a single `OtelBootstrap` startup entrypoint, a `withSpan(name, attrs) { ... }` helper, a `UserIdHasher.hash(uuid)` helper, and a `SpanRecorder` test fixture published via `java-test-fixtures`. Vendor-specific code (OTLP exporter, Grafana Cloud auth) lives entirely inside `:infra:otel` so a vendor swap (Open Decision #12) is a within-module change.
- **OTel SDK initialization at Ktor startup** with auto-instrumentation for Ktor HTTP server, the JDK / CIO HTTP client used by `:backend:ktor`'s outbound integrations (Supabase REST, Supabase Realtime publish, future Resend), Postgres JDBC via HikariCP, and Redis Lettuce. Initialization is idempotent and degrades cleanly to a no-op exporter when secrets are absent (local dev, unconfigured staging). The bootstrap reads slot-name strings via `secretKey(env, name)` and resolves their values via the existing `SecretResolver.resolve(name)` helper (matches the established two-step pattern in [`backend/ktor/.../config/Secrets.kt`](../../../backend/ktor/src/main/kotlin/id/nearyou/app/config/Secrets.kt)).
- **Mandatory spans** per [`docs/04-Architecture.md:397`](../../../docs/04-Architecture.md): HTTP request on Ktor (root span), Supabase API calls (REST + Realtime broadcast publish), Redis calls (rate-limit Lua scripts + cache reads — covered by the auto-instrumented Lettuce surface), PostGIS spatial queries (covered by auto-instrumented JDBC), FCM Admin SDK send (the local span only — see Deferrals below for cross-service `traceparent` propagation). Auto-instrumentation covers the bulk; a single manual `withSpan` wrap on the Supabase Realtime publish call is the only currently-shipped manual-span site this change introduces. Resend HTTP and CF Images calls are in the canonical doc but their producing modules don't exist yet (Resend unscaffolded; CF Images gated by `image_upload_enabled=FALSE` per Phase 4 §16) — those modules' future changes ADOPT this foundation, they don't re-introduce it.
- **Mandatory attributes** per [`docs/04-Architecture.md:398`](../../../docs/04-Architecture.md): `user.id` (SHA-256 hashed, first 16 hex chars, via `UserIdHasher.hash(...)`) when the request is authenticated against a `UserPrincipal`-backed identity (NOT applicable to `/internal/*` OIDC service-account auth — see Deferrals); `endpoint` and `http.route` (Ktor route pattern, NOT raw URL — query strings + path params stripped); `db.statement` (parameterized only, via the JDBC instrumentation's statement-sanitization setting); `supabase.realtime.channel` (channel name `chat:{conversation_id}` — the conversation_id is a primary key, not user-PII, so it is acceptable in span attributes); `cloud.region` (OTel semconv name; sourced from the GCP metadata server at `http://metadata.google.internal/computeMetadata/v1/instance/region`, fetched once at bootstrap and cached as a resource attribute — the canonical doc shorthand `geo.cloud_region` is amended to `cloud.region` via a `FOLLOW_UPS.md` entry).
- **Forbidden attributes** (defense-in-depth — captured as a positive requirement so the spec encodes the negative space): NEVER attach raw `user_id` UUIDs, raw client IPs (`CF-Connecting-IP` value), the auto-instrumentation's `client.address` / `net.peer.ip` / `net.sock.peer.addr` / `http.client_ip` (the Ktor server instrumentation's default attributes that would carry the Cloudflare-edge peer IP), raw JWT/refresh tokens, raw JWT claims (`sub`, `aud`, `iss`), raw bearer values, JWKS contents, raw post content, raw chat message content, raw search queries, raw Supabase admin keys, raw Redis cluster connection strings (the Lettuce auto-instrumentation's `db.connection_string` carries the userinfo password — the implementation MUST drop or sanitize the attribute), and raw `actual_location` GIS coordinates. The truncated SHA-256 token correlation id pattern from [`internal-endpoint-auth/spec.md:18`](../../specs/internal-endpoint-auth/spec.md) is the canonical anonymization shape.
- **W3C Trace Context propagation** on outbound HTTP from `:backend:ktor`: auto-instrumentation handles propagation for the Ktor / JDK HTTP client used by Supabase REST + Supabase Realtime publish. Cross-service propagation to FCM (which uses its own internal HTTP transport inside `:infra:fcm`) is **deferred** — see Deferrals.
- **Sampling profile** per [`docs/05-Implementation.md:2042`](../../../docs/05-Implementation.md):
  - `dev`: head 100% (`Sampler.alwaysOn()`; OTLP exporter target = LoggingSpanExporter / no-op when secrets absent).
  - `staging`: head 100% (`Sampler.alwaysOn()`; Phase 2 §14 benchmark surface).
  - `production`: 10% base ratio (`ParentBased(TraceIdRatioBased(0.1))`). **No force-keep promotion** — the canonical "100% errors + 100% slow" target requires OTel Collector tail sampling to preserve trace_id linkage, which is deferred to a follow-up change. MVP production accepts that 90% of healthy spans drop; structured JSON logging at 100% retention via Cloud Logging continues to be the authoritative surface for incident replay until the Collector lands.
- **Exporter wiring**: OTLP/HTTP to Grafana Cloud Tempo as the recommended default per Open Decision #12 ([`docs/08-Roadmap-Risk.md:586`](../../../docs/08-Roadmap-Risk.md)). Endpoint + token sourced via `secretKey(env, name)` slot-name derivation + `SecretResolver.resolve(name)` value lookup, against four new GCP Secret Manager slots: `otel-grafana-otlp-endpoint` + `otel-grafana-otlp-token` (production) and `staging-otel-grafana-otlp-endpoint` + `staging-otel-grafana-otlp-token` (staging). When EITHER value lookup returns null, the exporter falls back to the no-op `LoggingSpanExporter` (DEBUG severity, dropped by default Logback) and emits exactly one INFO startup line.
- **Replaces the WARN-only carve-out** in `chat-realtime-broadcast`: the line-228 explicit deferral is dropped; the publish call SHALL be wrapped in `withSpan("chat.realtime.publish", ...)`; on publish failure, the span records a status `ERROR` + an event matching the existing WARN log's `event` and `error_class` fields. Crash-window contract preserved.
- **Adds WARN-log↔span-event pairing** to `fcm-push-dispatch`: each of the 5 WARN-log scenarios (INVALID_ARGUMENT, retryable, unknown, per-token partial-failure, dispatch-after-shutdown) SHALL ALSO record a matching span event on the surrounding `withSpan("fcm.dispatch", ...)` span. Pair is best-effort — span-recording failure SHALL NOT block dispatch.
- **Intentionally NOT modified**: `internal-endpoint-auth`. The OIDC auth path's WARN logs already carry the truncated-SHA-256 token correlation id pattern. The cross-cutting Ktor server-span requirement covers `/internal/*` server spans (route pattern, status code) automatically. **The `user.id` attribute does NOT apply to `/internal/*` service-account auth** — defining a `service.account.id` attribute shape for `/internal/*` is deferred to a follow-up change (`internal-endpoint-auth-otel-attributes`).
- **Version Pinning Decisions Log entries** in [`docs/09-Versions.md`](../../../docs/09-Versions.md) for each new dependency.

## Deferrals (tracked in [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md))

These are NOT in scope of this change but are explicitly tracked so the trail is auditable:

1. **`observability-otel-collector-tail-sampling`**: deploys an OTel Collector and reworks production sampling to add the canonical "100% errors + 100% slow" promotion via Collector tail sampling — which is the only correct way to preserve trace_id linkage on force-keep. The naive SDK-level approach (secondary `BatchSpanProcessor` with a force-keep `SpanProcessor`) loses correlation because secondary-pipeline spans are detached from the original trace.
2. **`observability-otel-fcm-traceparent`**: refactors `:infra:fcm`'s `buildFcmComposite(...)` to surface `FirebaseOptions.Builder.setHttpTransport(...)`, ships a `TracingHttpTransport` wrapper in `:infra:otel`, and modifies the `fcm-push-dispatch` spec to require unconditional `traceparent` injection on the FCM Admin SDK's outbound HTTPS request. The local `withSpan("fcm.dispatch", ...)` lands in this change; cross-service propagation lands in the follow-up.
3. **`observability-otel-pii-disclosure`**: docs-only amendment to [`docs/06-Security-Privacy.md:308`](../../../docs/06-Security-Privacy.md) third-party processor list adding Grafana Cloud + UU PDP article 20–22 disclosure language; corresponding Privacy Policy update. Trace data carries pseudonymous personal data (truncated SHA-256 of user UUIDs); UU PDP disclosure obligations apply.
4. **`internal-endpoint-auth-otel-attributes`**: defines a sanctioned `service.account.id` attribute shape (truncated SHA-256 of OIDC `sub`) for `/internal/*` requests so the WARN-log token correlation id pattern extends to span attributes; modifies the `internal-endpoint-auth` spec.
5. **`observability-otel-attribute-detekt-rule`**: ships the Detekt rule enforcing the forbidden-attributes contract once writer-surface stabilizes (per design § D10).
6. **Rate-limit Lua wrap as a manual span site**: the design + spec list it but no scenario / task scaffolds it; auto-instrumented Lettuce spans cover the Redis layer adequately for MVP. Add manual wrapping if Phase 2 §14 benchmark surfaces specific latency questions Lettuce auto-instrumentation can't answer.

## Capabilities

### New Capabilities

- `observability-otel-foundation`: OpenTelemetry tracing foundation for `:backend:ktor` — initialization contract, mandatory spans, mandatory + forbidden attributes, W3C propagation for the auto-instrumented HTTP client surface, `ParentBased(TraceIdRatioBased(0.1))` production sampling, OTLP exporter wiring with no-op fallback, the `withSpan` helper, the `UserIdHasher` helper, and the `SpanRecorder` test fixture.

### Modified Capabilities

- `chat-realtime-broadcast`: drop the line-228 explicit deferral; replace with a positive pairing requirement — the publish call records a span; on publish failure, the span records an event with `event = "chat_realtime_publish_failed"` and `error.type` matching the WARN log's `error_class`. Crash-window contract preserved.
- `fcm-push-dispatch`: each of the 5 WARN-log scenarios SHALL also record a span event with the matching `event` attribute on the surrounding `"fcm.dispatch"` span. Pair is best-effort. The cross-service `traceparent` injection requirement is **NOT** added in this change (see Deferrals #2).

## Impact

- **Code**: new `:infra:otel` module with ~7 source files (`OtelBootstrap`, `OtelConfig`, `SamplingProfile`, `WithSpan`, `UserIdHasher`, plus the `testFixtures`-published `SpanRecorder`); Ktor startup wiring in `Application.module()`; HikariCP wrap at the existing `DataSourceFactory` site; Lettuce wrap at the two `RedisClient.create(url)` sites in [`infra/redis/.../RedisModule.kt`](../../../infra/redis/src/main/kotlin/id/nearyou/app/infra/redis/RedisModule.kt); Ktor server interceptor for the manual `Ktor` instrumentation if the `opentelemetry-instrumentation-ktor` library is unavailable for Ktor 3.4.x; manual `withSpan` wrap of the Supabase Realtime publish call inside `publishBroadcast(...)` at [`backend/ktor/.../chat/ChatRoutes.kt`](../../../backend/ktor/src/main/kotlin/id/nearyou/app/chat/ChatRoutes.kt); manual `withSpan` wrap of the FCM dispatch call inside [`infra/fcm/.../FcmDispatcher.kt`](../../../infra/fcm/src/main/kotlin/id/nearyou/app/infra/fcm/FcmDispatcher.kt) (local span only).
- **APIs**: zero contract change. Instrumentation is observable side-effect only.
- **Schema**: zero migrations.
- **Dependencies**: new pins in [`gradle/libs.versions.toml`](../../../gradle/libs.versions.toml) for the OTel SDK + exporter + JDBC + Lettuce instrumentation BOM. Each pin carries a row in [`docs/09-Versions.md`](../../../docs/09-Versions.md).
- **Mobile**: zero change.
- **Secrets**: 4 new GCP Secret Manager slots (`otel-grafana-otlp-endpoint`, `otel-grafana-otlp-token`, `staging-otel-grafana-otlp-endpoint`, `staging-otel-grafana-otlp-token`).
- **Cost**: per [`docs/01-Business.md:380-381`](../../../docs/01-Business.md), Grafana Cloud at Phase 2 traffic is 0 IDR (free tier sufficient); 0.2% of revenue at scale per [`docs/01-Business.md:454-455`](../../../docs/01-Business.md).
- **Out-of-scope (Non-goals — also captured in design.md)**:
  - Mobile-side OTel (KMP). Sentry KMP already covers crash + error reporting.
  - OTel Logs SDK / Metrics SDK. Structured JSON logging + GCP Cloud Monitoring + Supabase dashboards already cover those surfaces.
  - Force-keep promotion ("100% errors + 100% slow") — deferred to `observability-otel-collector-tail-sampling`.
  - Cross-service `traceparent` propagation to FCM — deferred to `observability-otel-fcm-traceparent`.
  - UU PDP disclosure for Grafana Cloud — deferred to `observability-otel-pii-disclosure` (docs-only follow-up).
  - `service.account.id` shape for `/internal/*` requests — deferred to `internal-endpoint-auth-otel-attributes`.
  - Detekt rule for forbidden attributes — deferred to `observability-otel-attribute-detekt-rule` per design § D10.
  - Vendor-final decision (Grafana Cloud vs Honeycomb vs Cloud Trace, Open Decision #12). This change ships Grafana Cloud as the recommended default.
  - Phase 2 §14 benchmark itself. This change unblocks it (staging at head-100% sampling) but doesn't run it.
