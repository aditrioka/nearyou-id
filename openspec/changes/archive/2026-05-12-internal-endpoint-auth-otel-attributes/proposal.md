## Why

The `observability-otel-foundation` capability (PR [#66](https://github.com/aditrioka/nearyou-id/pull/66), 2026-05-07) explicitly carved out `/internal/*` server spans from its `user.id` attribute requirement: per [`observability-otel-foundation/spec.md:152`](../../../openspec/specs/observability-otel-foundation/spec.md), *"`user.id` SHALL NOT be set for `/internal/*` requests authenticated via Cloud Scheduler service-account OIDC — those requests have no `users` row to hash. A sanctioned `service.account.id` shape (truncated SHA-256 of the OIDC `sub` claim) is deferred to the `internal-endpoint-auth-otel-attributes` follow-up; until that ships, `/internal/*` server spans carry `http.route` + `http.status_code` only (no principal-correlation attribute)."*

Today an operator looking at a slow `/internal/unban-worker` span in Grafana Tempo cannot group / filter / query-by-principal — the only correlation surface is via Cloud Logging cross-link (the WARN-log token correlation id at [`internal-endpoint-auth/spec.md:20`](../../../openspec/specs/internal-endpoint-auth/spec.md), which uses `SHA-256(raw token bytes)` per token, NOT per principal). For successful invocations there is no principal-correlation surface in the trace surface at all. As more `/internal/*` workers ship (privacy-flip, hard-delete, FCM cleanup, notifications purge, moderation archival per `docs/08-Roadmap-Risk.md` Phase 3.5), the gap compounds: every new worker will produce slow spans with no per-principal grouping, defeating the value of the trace surface for `/internal/*` debugging.

This change closes the carve-out by adding the sanctioned `service.account.id` attribute. Direct trigger: [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md) entry `internal-endpoint-auth-otel-attributes` (status: open) — its trigger condition was met when `observability-otel-foundation` shipped. The 2026-05-10 audit blurb at the top of `FOLLOW_UPS.md` classifies this entry as one of the "7 OpenSpec-shaped pending promotion via `/next-change`" entries.

## What Changes

- **NEW** `ServiceAccountIdHasher.hash(sub: String): String` in `:infra:otel` (sibling of [`UserIdHasher`](../../../infra/otel/src/main/kotlin/id/nearyou/app/infra/otel/UserIdHasher.kt) and [`IpHasher`](../../../infra/otel/src/main/kotlin/id/nearyou/app/infra/otel/IpHasher.kt)). Returns the first 16 hex chars of `SHA-256(sub.toByteArray(StandardCharsets.UTF_8))` — same output shape as the existing two helpers, keeping the operator mental model unified across user / IP / service-account correlation IDs. Defensive guard `require(sub.isNotBlank())` mirroring `IpHasher.hash`. Place at `infra/otel/src/main/kotlin/id/nearyou/app/infra/otel/ServiceAccountIdHasher.kt`.
- **NEW** attribute writer wired into the `InternalEndpointAuth` Ktor plugin (or its successor handler — implementation phase confirms the precise integration point per the existing `:backend:ktor` `internal` package wiring; the canonical placement is the same code path that sets the verified-principal context AFTER successful OIDC verification and BEFORE handler dispatch). The writer SHALL set `service.account.id = ServiceAccountIdHasher.hash(claims.sub)` on the active Ktor server span. Failure to access the active span (e.g., OTel SDK not initialised in tests) MUST short-circuit silently — the attribute write is best-effort observability, NOT a blocking auth gate.
- **MODIFY** `internal-endpoint-auth` capability — ADD requirement: every successful `/internal/*` request SHALL produce a server span carrying `service.account.id`. Scenarios cover: hashed-shape correctness, value derived from the OIDC `sub` claim (NOT raw token bytes — that's a different correlation id), no raw `sub` literal anywhere on the span, mutual-exclusion with `user.id` (a `/internal/*` span never carries `user.id`).
- **MODIFY** `observability-otel-foundation` capability — MODIFY the existing requirement "Mandatory span attributes SHALL be present on canonical span surfaces" (currently at [`observability-otel-foundation/spec.md:145`](../../../openspec/specs/observability-otel-foundation/spec.md)) to flip the `/internal/*` carve-out: `service.account.id` is now the sanctioned principal-correlation attribute for those requests. The MODIFIED requirement adds `service.account.id` to the bullet list of mandatory attributes (with the same conditional-on-authenticated wording, but with the principal-type discriminator: UserPrincipal → `user.id`; OIDC service-account → `service.account.id`). The "until that ships" prose at line 152 is removed.
- DELETE the `internal-endpoint-auth-otel-attributes` entry from [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md) at archive time — its scope is fully fulfilled by this change.

## Capabilities

### New Capabilities

(none — this change adds enforcement to two existing capabilities)

### Modified Capabilities

- `internal-endpoint-auth`: gains a new requirement that every successful `/internal/*` request produces a server span carrying `service.account.id` (16-hex truncated SHA-256 of the verified OIDC `sub` claim). Existing OIDC-verification, audience-check, and 401-response requirements are unchanged.
- `observability-otel-foundation`: MODIFIES the existing "Mandatory span attributes" requirement to add `service.account.id` as the sanctioned principal-correlation attribute for `/internal/*` requests, mutually-exclusive with `user.id`. The "deferred to follow-up" prose is replaced with a positive statement of the now-shipped contract. No other foundation requirements change.

## Impact

- **New code**: ~30 LOC for `ServiceAccountIdHasher.kt` + KDoc; ~80 LOC for the helper unit tests (mirror the `IpHasher` test shape including whitespace-not-trimmed, 1000-iteration random shape, parametrized blank-input set, plus a pinned-reference-hash assertion on `hash("test")` since the algorithm is contractually fixed by spec); ~10 LOC for the writer integration at `InternalEndpointAuth.kt:79` (one try/catch block, byte-for-byte mirror of `AuthPlugin.kt:113-120`); ~200–350 LOC for the integration test (initial estimate of ~50–100 was incorrect — there is NO existing `:backend:ktor` test using `KtorServerTelemetry`, so the new test must construct the OTel SDK + `SpanRecorder` + install `KtorServerTelemetry` itself; covers 8 spec scenarios across success / 401 / mutual-exclusion both directions / silent-fail SDK-uninitialised / silent-fail helper-throw / silent-fail FailingSpanProcessor / vendor-webhook opt-out).
- **Schema / APIs / Dependencies**: none. The OIDC `sub` claim is already extracted by the existing `OidcTokenVerifier` flow.
- **Privacy posture**: the OIDC `sub` claim is the GCP service-account principal identifier — a non-PII value (it identifies a service account, not a human end user). Per [`CLAUDE.md`](../../../CLAUDE.md) § Critical invariants → "Public repository posture": *"Slot names + GCP project IDs + service-account emails are non-sensitive."* Hashing it provides operational benefit (stable per-principal correlation in Tempo; cardinality bucketing) WITHOUT confidentiality intent — an attacker with read access to Tempo + the project name can dictionary-attack the 16-hex hash to recover the SA name in seconds (low-cardinality SA inventory + predictable naming convention). Acceptable because the SA name itself is non-sensitive per the posture statement. Setting the raw `sub` claim on the span would still be forbidden by the `OtelForbiddenAttributeRule` Tier 1 Group C entry (`jwt.sub`); the hashed-via-helper consumption is the sanctioned path.
- **Out of scope (explicit)**:
  - Span attribute for the WARN-log token correlation id (`SHA-256(raw token bytes)`). That correlation id serves a different purpose (per-token failure correlation) and lives only on the WARN log surface; it does NOT belong on the span (the span attribute is per-principal, the WARN log is per-token, and conflating the two would defeat the purpose of having both).
  - Adding `service.account.id` typo-defensive entries to `ForbiddenAttributeStripper.FORBIDDEN_KEYS` (e.g., `service_account_id`, `service.account_id`, `serviceAccount.id`). The runtime stripper already defensively guards `user_id`/`user_uuid`/`user.uuid` typos for the sanctioned `user.id` key (per `infra/otel/.../ForbiddenAttributeStripper.kt:103-107`). By the same precedent, sibling typo guards for the new sanctioned key would be cheap insurance. NOT included in this change because (a) the proposal's scope is the attribute-shape contract + the writer wiring; the runtime stripper's defensive entries are independent of the contract; (b) the Phase D round-1 security-and-invariant lens flagged this as a SUGGESTION (not a blocker); (c) splitting the typo-guard work into its own micro-change keeps the runtime-stripper change visible at PR-review time. A `FOLLOW_UPS.md` entry will be added at archive time if the round-2 review confirms the typo-guard work is desirable.
  - Adding `OtelForbiddenAttributeRule` typo-defensive entries to the compile-time lint surface (sibling concern to the runtime-stripper above). Same rationale: keep this change focused on the contract; defer the lint typo-guards to a follow-up if needed.
  - Choice of OTel semconv vs project-namespaced key. Resolved in `design.md` § Decision 1 — `service.account.id` is the canonical-doc-reserved name (per `observability-otel-foundation/spec.md:152`); aligning with that reservation is the right call here even though `service.account.id` is not in OTel semconv 1.30.0 stable (it is a project-namespaced extension under the standard `service.*` semconv root, which OTel allows for organisation-specific attributes).
  - Vendor-webhook routes (`/internal/revenuecat-webhook`, `/internal/csam-webhook`). Per the existing `internal-endpoint-auth` requirement "Plugin is mounted on the `/internal/*` subtree, with vendor-webhook opt-out", those routes do NOT use OIDC and therefore have NO `sub` claim to hash. Their server spans carry `http.route` + `http.status_code` only (consistent with the pre-this-change `/internal/*` baseline). A future change MAY add a vendor-specific principal-correlation attribute (`vendor.account.id` or similar) if regression risk surfaces.
  - Test fixtures asserting attribute absence on vendor-webhook routes. The mutual-exclusion scenario (`user.id` not set on `/internal/*` and vice versa) covers the principal-attribute discriminator at the OIDC-authenticated boundary; vendor-webhook coverage is independently owned by the existing webhook capabilities.
  - Cross-service `traceparent` propagation INTO `/internal/*` from Cloud Scheduler. Cloud Scheduler does not currently inject `traceparent` headers on its outbound HTTPS calls (verified via Cloud Scheduler docs at proposal time); the trace lineage from "Cloud Scheduler invokes endpoint" → "endpoint produces server span" begins at the server-span boundary. A future change MAY explore alternate correlation mechanisms (Cloud Scheduler request id custom header) if cross-service correlation becomes operationally important.
