## Why

1:1 chat is the next sequential Phase 2 capability per [`docs/08-Roadmap-Risk.md`](../../../docs/08-Roadmap-Risk.md) Phase 2 items 9–11, and it's the explicit motivator cited by the just-shipped [`fcm-push-dispatch`](../../specs/fcm-push-dispatch/spec.md) (PR [#60](https://github.com/aditrioka/nearyou-id/pull/60)) — every "you got a like / reply / follow" event today reaches the user only via the in-app `notifications` table because the FCM dispatcher has no `chat_message` emit-site to push. There is no chat schema yet: V2 already drafted a gated RLS policy on `realtime.messages` ([V2__auth_foundation.sql:58-78](../../../backend/ktor/src/main/resources/db/migration/V2__auth_foundation.sql)) that activates the moment `conversation_participants` exists, and the realtime token endpoint shipped in [`auth-realtime`](../../specs/auth-realtime/spec.md) already issues HS256 JWTs awaiting a real WSS consumer — but neither has anything to gate or token-against until the data plane lands. This change closes that gap by shipping the schema + REST API for 1:1 conversations + messages, deferring the Supabase Realtime broadcast publish, embedded-post send/render, admin redaction, Perspective API screening, per-conversation push batching, and the daily send-rate cap to follow-up changes so the first cut stays focused on a well-understood data plane.

## What Changes

- **NEW** Flyway V15 migration creating three tables matching the canonical schema in [`docs/05-Implementation.md` § Direct Messaging Implementation](../../../docs/05-Implementation.md):
  - `conversations(id UUID PK DEFAULT gen_random_uuid(), created_at TIMESTAMPTZ DEFAULT NOW(), created_by UUID NOT NULL REFERENCES users(id) ON DELETE RESTRICT, last_message_at TIMESTAMPTZ NULL)`. 1:1-only for MVP — no `type` column; group chat is post-MVP per [`docs/02-Product.md` § Chat](../../../docs/02-Product.md). `last_message_at` is bumped by the chat-message send path (no DB trigger; Ktor sets it in the same transaction as the INSERT — keeps the trigger surface minimal and matches the canonical insert flow). The `created_by` FK uses `RESTRICT` to mirror the canonical `chat_messages.sender_id` semantics; hard-delete of a user with conversation history must go through the Phase 3.5 cleanup worker which will null-out PII columns before deletion. Documented in `design.md` § Decisions D2.
  - `conversation_participants(conversation_id UUID FK ON DELETE CASCADE, user_id UUID FK ON DELETE RESTRICT, slot SMALLINT NOT NULL CHECK (slot IN (1, 2)), joined_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), left_at TIMESTAMPTZ NULL, last_read_at TIMESTAMPTZ NULL, PRIMARY KEY (conversation_id, user_id))`. **Slot-based partial unique** `CREATE UNIQUE INDEX conv_slot_unique ON conversation_participants (conversation_id, slot) WHERE left_at IS NULL` enforces the 2-active-participant cap — the slot column is the canonical mechanism per [`docs/05-Implementation.md`](../../../docs/05-Implementation.md). The participant-insert flow takes a Postgres advisory lock keyed on the conversation id (`pg_advisory_xact_lock(hashtext(:conversation_id::text))`) to serialize concurrent slot races within an existing conversation. Additionally, the create-or-return endpoint takes a separate advisory lock keyed on the canonical-ordered user pair to serialize concurrent first-creation attempts for the same pair — see `design.md` § Decisions D1. `last_read_at` ships now (column-only) so the Phase 2 follow-up unread-count work has the storage; this change does not write or read it. Indexes: `conversation_participants_user_active_idx (user_id) WHERE left_at IS NULL` for "list my conversations," `conversation_participants_conversation_idx (conversation_id)` for participant lookup.
  - `chat_messages(id UUID PK DEFAULT gen_random_uuid(), conversation_id UUID FK ON DELETE CASCADE, sender_id UUID NOT NULL FK ON DELETE RESTRICT, content VARCHAR(2000) NULL, embedded_post_id UUID NULL FK ON DELETE SET NULL, embedded_post_snapshot JSONB NULL, embedded_post_edit_id UUID NULL, created_at TIMESTAMPTZ DEFAULT NOW(), redacted_at TIMESTAMPTZ NULL, redacted_by UUID NULL, redaction_reason TEXT NULL)`. **Atomicity CHECK** `((redacted_at IS NULL AND redacted_by IS NULL AND redaction_reason IS NULL) OR (redacted_at IS NOT NULL AND redacted_by IS NOT NULL))` per canonical (3-column coupling; `redaction_reason` is optional free-text but only when redacted). **Empty-message CHECK** `(content IS NOT NULL OR embedded_post_id IS NOT NULL OR embedded_post_snapshot IS NOT NULL)`. Indexes: `chat_messages_conv_idx (conversation_id, created_at DESC)`, `chat_messages_sender_idx (sender_id, created_at DESC)`, `chat_messages_redacted_idx (redacted_by, redacted_at DESC) WHERE redacted_at IS NOT NULL`. **Two FK columns ship without their FK constraint** — both documented via `COMMENT ON COLUMN` SQL statements following the V9 deferred-FK pattern at [V9__reports_moderation.sql:72-73, 110-111](../../../backend/ktor/src/main/resources/db/migration/V9__reports_moderation.sql): (a) `redacted_by` documented as `'FK to admin_users(id) ON DELETE SET NULL — deferred to the Phase 3.5 admin-users migration.'` to be added with the Phase 3.5 admin schema; (b) `embedded_post_edit_id` documented as `'FK to post_edits(id) ON DELETE SET NULL — deferred to the post-edit-history change which ships post_edits.'` to be added when `post_edits` lands. `post_edits` does NOT exist at V14 — verified via `grep -rn "CREATE TABLE post_edits" backend/ktor/src/main/resources/db/migration/` returning no matches; the table is canonical Phase 4 schema per [`docs/08-Roadmap-Risk.md:102`](../../../docs/08-Roadmap-Risk.md). Both columns are typed `UUID NULL`.
- **NEW** REST endpoints in `:backend:ktor`:
  - `POST /api/v1/conversations` body `{ recipient_user_id }` — create-or-return 1:1 conversation. Self-DM (recipient == caller) rejected `400 Bad Request` BEFORE the user-pair lock is taken. Resolves canonical-ordered pair `(LEAST(caller, recipient), GREATEST(caller, recipient))`, takes the user-pair advisory lock, looks up an existing 1:1 conversation between the two via `conversation_participants`, creates one if absent (INSERT into `conversations`, two INSERTs into `conversation_participants` with `slot=1` for the creator and `slot=2` for the recipient under the conversation-id advisory lock), releases lock. Recipient-existence check reads RAW `users` (NOT `visible_users`) so the response does not leak shadow-ban state via a 201/404 differential — the canonical shadow-ban contract requires that non-banned third parties cannot detect a user's banned state by probing endpoints. Bidirectional block check rejects with `403 Forbidden` (matches the canonical "blocked user cannot initiate a DM" semantics in [`docs/02-Product.md:234`](../../../docs/02-Product.md)). Idempotent on duplicate creation attempt — returns the existing conversation with `200 OK` (vs `201 Created` on first creation; status-code differential is the only observable difference and is part of the contract).
  - `GET /api/v1/conversations` — list active conversations (`left_at IS NULL` on the caller's row), ordered by `last_message_at DESC NULLS LAST, created_at DESC` for newly-created-empty conversations, cursor-paginated. Other-participant profile read via **LEFT JOIN `visible_users`** with `COALESCE` to placeholder columns (`username = 'akun_dihapus'`, `display_name = 'Akun Dihapus'`, `is_premium = FALSE`) so a shadow-banned partner's conversation still surfaces in the list with masked profile fields — the canonical invisible-actor model: the partner's chat-list visibility is preserved, but their profile is masked. **No bidirectional block-exclusion** on the list query — per the canonical block-aware-chat contract at [`docs/02-Product.md:234`](../../../docs/02-Product.md) "Existing conversations remain visible in history," the list view is symmetric with the messages view (both surface conversations post-block; only create + send enforce the 403 bidirectional-block check). The query site carries the annotation `// @allow-no-block-exclusion: chat-history-readable-after-block` to suppress `BlockExclusionJoinRule` on the partner-lookup join, matching the same carve-out applied to list-messages and send-message participant queries.
  - `GET /api/v1/chat/{conversation_id}/messages` — list messages in a conversation matching canonical [`docs/05-Implementation.md:1215`](../../../docs/05-Implementation.md). Caller MUST be active participant (`participant.left_at IS NULL`); 403 otherwise. Cursor pagination via `(created_at DESC, id DESC)`, default 50/page, hard cap 100/page. **Shadow-ban filter on the read path**: the query returns chat_messages rows where `(sender_id = :viewer OR NOT EXISTS shadow-ban-on-sender)` — a viewer ALWAYS sees their own messages (own-content carve-out), but a non-banned viewer never sees a shadow-banned sender's messages. This is the canonical filter pattern symmetric with `visible_posts`/`visible_users`. Redacted messages return their row with `content` masked to NULL and a `redacted_at` timestamp surfaced (so the client renders `"Pesan ini telah dihapus oleh moderator."` per [`docs/05-Implementation.md:1299`](../../../docs/05-Implementation.md)). Reads from raw `chat_messages` are explicitly allowed in this Repository own-content path per [`CLAUDE.md` § Critical invariants](../../../CLAUDE.md) carve-out; the shadow-ban filter on `sender_id` is the same shape as the `visible_posts` view but inlined per-query because the own-content carve-out (sender = viewer) is per-row. The `BlockExclusionJoinRule` carve-out for chat-history-readable-after-block is documented in `design.md` § D9 and applied via the source annotation `// @allow-no-block-exclusion: chat-history-readable-after-block`.
  - `POST /api/v1/chat/{conversation_id}/messages` body `{ content }` matching canonical [`docs/05-Implementation.md:1198`](../../../docs/05-Implementation.md). Active-participant check (403 otherwise), bidirectional block check via the canonical query in [`docs/05-Implementation.md:1304-1308`](../../../docs/05-Implementation.md) (403 with user-facing `"Tidak dapat mengirim pesan ke user ini"` if either direction in `user_blocks`), 2000-char length guard via the existing content-length middleware (newly registered for the chat path with `maxChars = 2000`). On success: INSERT chat_messages row, UPDATE `conversations.last_message_at = NOW()` in the same transaction, return the inserted row. **Shadow-banned senders are allowed to send** (own-content carve-out — the row persists, the sender sees normal app behavior; non-banned recipients silently never see the row via the `GET /messages` filter described above). The send path's participant lookup is also covered by the chat-history-readable-after-block carve-out for `BlockExclusionJoinRule`. Any `embedded_post_id`, `embedded_post_snapshot`, OR `embedded_post_edit_id` field present in the request body is silently ignored for this cut (deferred to `chat-embedded-posts` follow-up); a structured WARN log per ignored field with the resulting message-id and `event = "chat_send_embedded_field_ignored"` is emitted.
- **MODIFIED** [`auth-realtime`](../../specs/auth-realtime/spec.md) capability — V15 SHALL CREATE the `participants_can_subscribe` policy on `realtime.messages` directly (V2's gated DO block did not install the policy because `conversation_participants` did not exist at V2-time, and Flyway is forward-only). The CREATE POLICY block in V15 SHALL match V2's structure EXCEPT the `is_shadow_banned` clause on the subscriber-side check (V2 lines 81-84) is REMOVED — shadow-banned users SHALL be allowed to subscribe to their own conversation realtime channels, consistent with the D9 invisible-actor model committed in this change. The `auth-realtime` capability gains an ADDED scenario verifying the policy is installed by V15 with the corrected definition. The token-issuance contract stays as-is. The policy installation + the corrected behavior are verified by the new RLS test set in this change (mandatory per `CLAUDE.md` invariant on RLS policy changes).

## Capabilities

### New Capabilities
- `chat-conversations`: 1:1 conversation + chat-message data plane — `conversations` / `conversation_participants` / `chat_messages` schema with slot-based 2-participant cap (partial unique on slot + per-conversation advisory lock for slot assignment + per-user-pair advisory lock for create-or-return), block-aware send + create-conversation, atomicity + empty-message CHECKs, soft-leave participants, sender FK ON DELETE RESTRICT (hard-delete handled by Phase 3.5 cleanup worker), redacted-message render contract. REST endpoints for create-or-return conversation, list conversations, list messages (cursor-paginated, redaction-aware), and send message (block-aware, length-guarded).

### Modified Capabilities
- `auth-realtime`: cross-reference update only — add a scenario asserting that the V2-drafted RLS policy on `realtime.messages` activates once V15 creates `conversation_participants`. The token endpoint contract itself does not change; this change documents the consumer side of the token contract that until now was a pure forward-reference.

## Impact

- **Affected modules**: `:backend:ktor` (new chat routes + repository + content-length middleware registration); `backend/ktor/src/main/resources/db/migration/V15__chat_foundation.sql` (new). No new Gradle module needed (chat lives in `:backend:ktor` per the same shape as `posts`, `replies`, `reports`).
- **Affected APIs**: 4 new REST endpoints. `POST /api/v1/conversations` + `GET /api/v1/conversations` (conversation-level resource); `GET /api/v1/chat/{conversation_id}/messages` + `POST /api/v1/chat/{conversation_id}/messages` (message-level resource matching canonical [`docs/05-Implementation.md:1198,1215`](../../../docs/05-Implementation.md)). The two-prefix shape is canonical; aligning end-to-end avoids needing to change paths in a follow-up.
- **Affected schema**: 3 new tables, 6 new indexes (1 partial unique on slot, 2 on `conversation_participants` for read paths, 3 on `chat_messages` for read + admin), 4 CHECKs (slot enum, content-or-embed empty-message guard, 3-column redaction atomicity, slot ∈ {1,2}). Adds two `COMMENT ON COLUMN` statements documenting the deferred FK constraints (matching the V9 pattern at [V9__reports_moderation.sql:72-73, 110-111](../../../backend/ktor/src/main/resources/db/migration/V9__reports_moderation.sql)): one on `chat_messages.redacted_by` (Phase 3.5 admin-users) and one on `chat_messages.embedded_post_edit_id` (post-edit-history change). **V15 SHALL CREATE the `realtime.messages` RLS policy directly** — V2's policy creation is gated on `conversation_participants` existing AND was a one-time DO block when V2 originally migrated (which evaluated false because `conversation_participants` did not exist at V2-time, and Flyway is forward-only so V2 will never re-run); the proposal's earlier claim of "activates as a side-effect" was incorrect. V15 includes the canonical policy creation with the **corrected** definition (the `is_shadow_banned` clause on the SUBSCRIBER-side check from V2 lines 81-84 is REMOVED — it contradicts the D9 invisible-actor model; shadow-banned users SHALL be allowed to subscribe to their own conversation realtime channels per `design.md` § D9).
- **Affected tests**: new `ChatFoundationRouteTest` + repository tests + RLS test set ("JWT sub not in `users` deny" per `CLAUDE.md` invariant + non-participant deny + `left_at` participant deny + malformed topic deny + SQL-injection topic deny — already drafted in V2 against fixture data, now exercising real schema). Block-exclusion join coverage on the partner-lookup path. Slot-race concurrency test (10 concurrent `POST /api/v1/conversations` for the same pair → exactly one row pair). Empty-message CHECK test. Redaction atomicity CHECK test.
- **Affected lint rules**: existing `BlockExclusionJoinRule` enforces the bidirectional block join on the conversation-list partner lookup — no new rule needed. For the `GET /messages` and `POST /messages` queries that touch `chat_messages` directly, the canonical "block-aware send + block-aware list" contract per [`docs/02-Product.md:234`](../../../docs/02-Product.md) is asymmetric: send rejects with 403 if blocked, but list-history-readable-after-block must NOT filter messages out of an existing conversation (per the canonical product spec — both parties retain history visibility). The lint rule's bidirectional NOT-IN would over-filter the list path. This change introduces a per-call-site annotation `// @allow-no-block-exclusion: chat-history-readable-after-block` (matching the existing project annotation convention from `// @allow-username-write` and `// @allow-privacy-write`); the rule is updated to recognize the annotation. The annotation is applied to the list-messages and send-message participant-lookup query sites; the send-message endpoint additionally runs an EXPLICIT bidirectional `user_blocks` lookup (canonical query at [`docs/05-Implementation.md:1304-1308`](../../../docs/05-Implementation.md)) to enforce the 403 send-block contract — the lint annotation only suppresses the AUTOMATIC NOT-IN-`user_blocks` join enforcement, not the explicit query in the handler. Hash-tag Redis keys are not used by this change (no caching layer in scope; Redis-backed unread counters are post-MVP).
- **Affected secrets**: none. Supabase JWT secret already wired by `auth-realtime`. No new secret slots.
- **Affected docs**: [`docs/05-Implementation.md`](../../../docs/05-Implementation.md) chat schema + REST contracts already canonical — proposal aligns with them. No doc amendments required at proposal time. If the reconciliation pass surfaces any divergence, the proposal will track it (Phase B step 7 gate).
- **Backwards compatibility**: zero. Net-new schema + net-new endpoints. The V2-drafted RLS policy activating is a behavior surface that comes online for the first time, not a change to existing surface.
- **Out of scope (each is a follow-up change):**
  - Supabase Realtime broadcast publish from Ktor (`chat-realtime-broadcast` follow-up — Phase 2 #9 realtime layer).
  - Daily send-rate cap (Free 50/day per [`docs/02-Product.md:318`](../../../docs/02-Product.md), Premium unlimited) — `chat-rate-limit` follow-up using the shared `rate-limit-infrastructure` capability shipped in `like-rate-limit`. Same shape as the existing like-rate-limit + reply-rate-limit changes.
  - Embedded post send/render path (`chat-embedded-posts` follow-up — schema columns ship now, but client-facing send/render is deferred).
  - Admin chat-message redaction endpoint (Phase 3.5 admin-panel territory; the schema columns and atomicity CHECK ship now).
  - Per-conversation FCM push batching (Phase 2 #11 follow-up; depends on realtime path landing first).
  - `chat_message` notification emit-site (separate change once realtime broadcast lands; FCM dispatcher in PR [#60](https://github.com/aditrioka/nearyou-id/pull/60) will pick it up automatically when emitted).
  - Perspective API screening of chat content (`perspective-api-moderation` follow-up — applies across posts/replies/chat as one cross-cutting change).
  - Read-receipts (`last_read_at` column ships but is unused by this change), typing indicators, message reactions, unread counters — post-MVP per [`docs/02-Product.md`](../../../docs/02-Product.md) § Chat.
  - Hourly conversation-list cache, Redis-backed unread counters — Month 6+ scaling territory.
