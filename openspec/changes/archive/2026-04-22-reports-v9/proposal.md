## Why

V7 (`post-likes-v7`) and V8 (`post-replies-v8`) shipped the engagement primitives but explicitly deferred the auto-hide mechanism that their `is_auto_hidden` columns exist to receive. `docs/08-Roadmap-Risk.md:150` is explicit: "rate limit + auto-hide trigger remain (deferred to rate-limit change + `reports` change)". The reports DDL is already drafted verbatim at `docs/05-Implementation.md:745-775`, the trigger semantics at `docs/05-Implementation.md:777-782`, the reply-side mirror at `docs/05-Implementation.md:737`, and the `moderation_queue` schema — which also consumes future Perspective API / admin-flag / CSAM / anomaly-detection / username-flagged triggers — at `docs/05-Implementation.md:789-816`. V9 is the natural next step: it wires the moderation layer that V4 `posts`, V7 `post_likes`, and V8 `post_replies` have each been column-reserving for.

## What Changes

- **V9 migration `V9__reports_moderation.sql`** verbatim-aligned with `docs/05-Implementation.md:745-816` — adds two tables:
  - `reports (id, reporter_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, target_type VARCHAR(16) CHECK IN ('post','reply','user','chat_message'), target_id UUID NOT NULL, reason_category VARCHAR(32) CHECK IN ('spam','hate_speech_sara','harassment','adult_content','misinformation','self_harm','csam_suspected','other'), reason_note VARCHAR(200), status VARCHAR(16) DEFAULT 'pending' CHECK IN ('pending','actioned','dismissed'), created_at, reviewed_at, reviewed_by UUID, UNIQUE(reporter_id, target_type, target_id))` + 3 indexes (`reports_status_idx`, `reports_target_idx`, `reports_reporter_idx`). Cascade on `reporter_id` matches the Data Export scope matrix note at `docs/05-Implementation.md:763-768` (user's submitted reports are included in their export, so cascade-delete is safe).
  - `moderation_queue (id, target_type VARCHAR(16), target_id UUID, trigger VARCHAR(32) CHECK IN ('auto_hide_3_reports','perspective_api_high_score','uu_ite_keyword_match','admin_flag','csam_detected','anomaly_detection','username_flagged'), status VARCHAR(16) DEFAULT 'pending' CHECK IN ('pending','resolved'), resolution VARCHAR(32) CHECK IN (...), priority SMALLINT DEFAULT 5, created_at, resolved_at, resolved_by UUID, notes TEXT, UNIQUE(target_type, target_id, trigger))` + 2 indexes (`moderation_queue_status_idx` on `(status, priority, created_at)`, `moderation_queue_target_idx` on `(target_type, target_id)`). Full enum lists ship at V9 even though V9 only writes `auto_hide_3_reports` rows — avoids an enum-widening migration when Phase 2 item 16 (Perspective API) and Phase 3.5 admin-flag / CSAM triggers land. The `username_flagged` enum value was introduced conceptually at `docs/05-Implementation.md:794-795` and `docs/05-Implementation.md:492` for the Phase 4 Premium username customization flow.
  - **Deferred-FK convention (new)**: `reports.reviewed_by` and `moderation_queue.resolved_by` are `UUID` columns **without** FK in V9 because `admin_users` ships in the Phase 3.5 admin-panel change. Column-level `COMMENT` on each records the deferred target (`admin_users(id) ON DELETE SET NULL`). This is the first time the pipeline deliberately defers an FK; the admin-users change will `ALTER TABLE ... ADD CONSTRAINT ... NOT VALID` then `VALIDATE`.
- **`POST /api/v1/reports`** — Bearer JWT required; body `{ target_type, target_id, reason_category, reason_note? }`; validate: `target_type` in the 4-value enum, `reason_category` in the 8-value enum, `reason_note` ≤ 200 chars (content-length middleware per Phase 1 item 20); **target existence check** against the appropriate table (`posts` / `post_replies` / `users` / `chat_messages`) — 404 on missing or soft-deleted target; **deliberately NOT block-exclusion-filtered** (a user may report content from a user they blocked or who blocked them; reporting a shadow-banned user's content also valid). Self-report (`target_type = 'user' AND target_id = :reporter_id`) rejected 400. Insert the row; 409 on UNIQUE `(reporter_id, target_type, target_id)` collision with `reports.duplicate` error code. Return 204 (no body; client already has the state it needs). Rate limit **10 submissions/hour/user** via existing `{scope:rate_report}:{user:<id>}` Redis hash-tag key — prevents report-bombing.
- **Auto-hide check (same DB transaction as the `reports` insert)** — immediately after the insert, `SELECT COUNT(DISTINCT r.reporter_id) FROM reports r JOIN users u ON u.id = r.reporter_id WHERE r.target_type = ? AND r.target_id = ? AND u.created_at < NOW() - INTERVAL '7 days' AND u.deleted_at IS NULL`. The account-age filter on reporters is the 3-unique-reporters-accounts->7-days rule per `docs/08-Roadmap-Risk.md:317` and `docs/05-Implementation.md:777`. If count ≥ 3:
  - `target_type = 'post'` → `UPDATE posts SET is_auto_hidden = TRUE WHERE id = ? AND deleted_at IS NULL` (idempotent; already-hidden row is a no-op).
  - `target_type = 'reply'` → `UPDATE post_replies SET is_auto_hidden = TRUE WHERE id = ? AND deleted_at IS NULL`.
  - `target_type IN ('user','chat_message')` → no column update (neither table has `is_auto_hidden`; the `moderation_queue` row is the sole admin-triage signal; chat-message redaction is a Phase 3.5 admin action — reports on chat_messages accumulate for admin review only).
  - In all 4 cases: `INSERT INTO moderation_queue (target_type, target_id, trigger) VALUES (?, ?, 'auto_hide_3_reports') ON CONFLICT (target_type, target_id, trigger) DO NOTHING` — idempotent via the schema's UNIQUE constraint; the same (target, trigger) pair is never enqueued twice.
- **Zero read-path changes to existing timelines and reply-list endpoints**. `visible_posts` (V4) already filters `is_auto_hidden = FALSE`, so nearby/following timelines automatically hide reported posts the moment the flag flips. GET `/api/v1/posts/{post_id}/replies` (V8) already filters `is_auto_hidden = FALSE OR author_id = :viewer`, so auto-hidden replies auto-vanish for non-authors. **No changes** to `nearby-timeline`, `following-timeline`, `post-likes`, `post-replies`, or `visible-posts-view` read contracts.
- **`RawFromPostsRule` + raw-table-read exception list** — ReportService's target-existence readers on `posts` / `post_replies` / `users` / `chat_messages` are legitimate point-lookups (by id, not listings; moderation-style access). Add ReportService to the rule's allow-list alongside the existing Repository own-content + admin-module exceptions. Positive + negative Detekt fixtures updated.
- **CI lint: `BlockExclusionJoinRule` is NOT modified**. Its protected-tables list already covers `posts` / `users` / `post_replies` / `chat_messages`, and ReportService does not emit listing queries (only point-lookup + INSERT), so the rule does not fire on V9 code.

## Capabilities

### New Capabilities
- `reports`: public Report submission capability — `POST /api/v1/reports` (JWT-required, reason-category + optional note, target existence check without block-exclusion filtering, 10/hour/user rate limit, self-report rejection, 409 duplicate handling); `reports` table schema + 3 indexes + UNIQUE `(reporter_id, target_type, target_id)`; reporter_id ON DELETE CASCADE from users; `reviewed_by` deferred-FK convention; auto-hide coupling (transactional count-distinct-reporters + conditional is_auto_hidden flip on post/reply + idempotent `auto_hide_3_reports` row in moderation_queue).
- `moderation-queue`: admin-triage capability — `moderation_queue` table schema + 2 indexes + UNIQUE `(target_type, target_id, trigger)`; full trigger enum (7 values) ships at V9 for forward-compatibility with Perspective API / admin flag / CSAM / anomaly / username-flagged write paths; full resolution enum (8 values) ships for the Phase 3.5 admin review UI; V9 writes only `auto_hide_3_reports` rows (the other 6 triggers are no-op writers until their respective features land); `resolved_by` deferred-FK convention mirrors `reports.reviewed_by`.

### Modified Capabilities
- `users-schema`: V9 adds one outgoing FK edge from `reports.reporter_id → users(id) ON DELETE CASCADE`. Matches V5 `user_blocks` / V6 `follows` / V7 `post_likes` CASCADE pattern (not the V4 `posts` / V8 `post_replies` RESTRICT pattern). Rationale is Data Export coverage — user's submitted reports are in their export, so hard-delete cascade is safe and matches privacy policy.
- `post-creation`: behavioral note — `posts.is_auto_hidden` is now actively written (by the V9 reports auto-hide trigger); V4 reserves the column, V4's `visible_posts` view already filters it, but no code path set the flag until V9. Read-path contract unchanged.
- `post-replies`: behavioral note — `post_replies.is_auto_hidden` is now actively written (by the V9 reports auto-hide trigger with `target_type = 'reply'`); V8 reserves the column and V8's reply-list endpoint already filters `is_auto_hidden = FALSE OR author_id = :viewer`; the author-bypass behavior is unchanged but now observable in practice. Rate-limit deferral note from V8 remains (20/day Free cap is a separate change).
- `migration-pipeline`: record V9; document the **fifth distinct dependency shape** — schema-level single-side CASCADE FK on V3 `users` (for `reports.reporter_id`) + TWO deferred FK columns targeting a not-yet-shipped `admin_users` table. First deliberate deferred-FK in the pipeline; the pattern (UUID column + `COMMENT ON COLUMN` recording the deferred target + Phase 3.5 `ADD CONSTRAINT ... NOT VALID` + `VALIDATE` follow-up) becomes the convention for any future column that must reference a downstream-introduced table.
- `visible-posts-view`: add ReportService to the `RawFromPostsRule` exception list (same structural list that already exempts Repository own-content paths and the admin module). No change to the view itself — `WHERE is_auto_hidden = FALSE` is exactly the contract V9 depends on.

  - **Rate limits on likes and replies** (Free 10/day likes, 20/day replies) — separate "rate-limit change" per the roadmap. V9 introduces only the report-submission rate limit (10/hour/user), which is anti-abuse for the reports endpoint itself, not a Phase 2 item 3 deferral.
  - **Admin review UI** reading `moderation_queue` — Phase 3.5 admin-panel change. V9 writes rows; no admin endpoint reads them yet.
  - **`post_auto_hidden` notification** (`docs/05-Implementation.md:861`) — depends on the notifications API change (Phase 2 item 5). V9's auto-hide path has a clearly-marked `// TODO: notifications-api-v??` stub at the flip point.
  - **Perspective API / UU ITE keyword / CSAM / anomaly / username_flagged writers** into `moderation_queue` — each has its own feature change. V9 just creates the table with its full enum list.
  - **`reviewed_by` / `resolved_by` FK to `admin_users`** — deferred to the Phase 3.5 admin-users migration (`ALTER TABLE ... ADD CONSTRAINT`).
  - **Report history endpoint** (user-facing list of "my submitted reports") — not in Phase 2 spec; user's export covers this via Data Export.
  - **Auto-unhide** on reporter revocation / false-positive resolution — Phase 3.5 admin-panel capability (admin sets `moderation_queue.resolution = 'keep'` and manually flips `is_auto_hidden = FALSE`).
  - **Mobile UI** for the reason-picker flow — tracked under Phase 3 mobile work.
  - **Block-exclusion filtering on target resolution** — deliberately omitted (reporting blocked/blocking users is valid behavior).
