## Why

The `content-moderation-keyword-lists` capability shipped at PR [#70](https://github.com/aditrioka/nearyou-id/pull/70) (2026-05-07) with the canonical contract at [`content-moderation-keyword-lists/spec.md:372`](../../../openspec/specs/content-moderation-keyword-lists/spec.md): every write-path handler that wires `TextModerator.moderate(...)` SHALL run the moderator AFTER existing length validation, rate-limit check, and (where applicable) block check, AND BEFORE the content INSERT. The contract is enforced today by THREE static-source-scan integration tests:

- [`PostCreationModerationIntegrationTest.kt:477`](../../../backend/ktor/src/test/kotlin/id/nearyou/app/post/PostCreationModerationIntegrationTest.kt) — `"POST /api/v1/posts handler source: length guard → moderator → INSERT call order"`
- [`PostRepliesModerationIntegrationTest.kt:406`](../../../backend/ktor/src/test/kotlin/id/nearyou/app/engagement/PostRepliesModerationIntegrationTest.kt) — `"ReplyService.post: visibility resolution → moderator → INSERT → notification emit call order"`
- [`ChatModerationIntegrationTest.kt:442`](../../../backend/ktor/src/test/kotlin/id/nearyou/app/chat/ChatModerationIntegrationTest.kt) — `"ChatRepository.sendMessage: block check → preInsertHookInTx → INSERT call order"`

These tests catch the regression at TEST time. A Detekt rule `ContentWriteRequiresModerationRule` would catch it at COMPILE time — defense-in-depth, sibling to the existing seven Detekt rules (`OtelForbiddenAttributeRule`, `RawFromPostsRule`, `BlockExclusionJoinRule`, `RedisHashTagRule`, `RawXForwardedForRule`, `RateLimitTtlRule`, `CoordinateJitterRule`).

Direct trigger: [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md) entry `content-write-moderation-detekt-rule` (status: open) — its trigger condition was met when `content-moderation-keyword-lists` shipped (the entry was deferred per task 10.4 decision gate at that time). The 2026-05-10 audit blurb classifies this as one of the "7 OpenSpec-shaped pending promotion via `/next-change`" entries.

**Honest impact framing** (per the FOLLOW_UPS entry's own assessment): this is a **belt-and-suspenders** addition. The static-source-scan tests are the canonical enforcement; the Detekt rule's net new value depends on how often new content-write paths are added. The case for shipping it now: (a) shifts enforcement from test-time to compile-time (faster developer feedback loop — IDE squiggles vs `:backend:ktor:test` run); (b) regression prevention for future contributors who add a new content-write path (e.g., a Phase 4 admin chat-redaction edit, or a future post-edit path) without realising the call-order requirement; (c) mirrors the recently-shipped lint-rule pattern (PR #99 `otel-attribute-lint-rule`, in-flight PR [#100](https://github.com/aditrioka/nearyou-id/pull/100) `rate-limit-ip-axis-method-lint-rule`) keeping the project Detekt-rule surface consistent.

## What Changes

- **NEW** `ContentWriteRequiresModerationRule.kt` in `lint/detekt-rules/src/main/kotlin/id/nearyou/lint/detekt/`. Fires on `KtCallExpression` invocations of canonical content-write surfaces — the JDBC `INSERT INTO posts (..., content, ...) VALUES (...)`, `INSERT INTO post_replies (..., content, ...)`, `INSERT INTO chat_messages (..., content, ...)` patterns OR the equivalent service-layer write methods (`PostRepository.create`, `ReplyService.post`, `ChatRepository.sendMessage`) — when the enclosing function does NOT also invoke `TextModerator.moderate(...)` BEFORE the write call. The rule's mechanism is PSI-walk: locate the candidate write-call expression, walk its containing function's PSI tree, search for a `TextModerator.moderate` invocation that PSI-precedes the write call. Implementation phase confirms the precise gate (likely a regex match on the SQL string literal or a callee-name match on the service method, depending on which surface is more robust).
- Register `ContentWriteRequiresModerationRule` in `lint/detekt-rules/src/main/kotlin/id/nearyou/lint/detekt/NearYouRuleSetProvider.kt` (the registration list currently has 7 rules; this would be the 8th — though [PR #100](https://github.com/aditrioka/nearyou-id/pull/100) is also adding the 8th, so whichever change merges second will need to rebase to add itself as the 9th).
- Wire into the project Detekt config — verify default-active behaviour matches the OtelForbiddenAttributeRule + RateLimitTtlRule + RedisHashTagRule + CoordinateJitterRule + RawXForwardedForRule precedent (no explicit enable line in [`backend/ktor/config/detekt/detekt.yml`](../../../backend/ktor/config/detekt/detekt.yml)).
- Ship `ContentWriteRequiresModerationLintTest` covering ~10 fixture scenarios:
  - **Positive (rule fires):** handler writes `INSERT INTO posts (..., content, ...)` without preceding `TextModerator.moderate` invocation; same for `post_replies`; same for `chat_messages`; service-method-form (`PostRepository.create(content, ...)` without preceding `moderate`).
  - **Negative (rule passes):** handler invokes `TextModerator.moderate(content)` THEN proceeds to INSERT; fixture path is canonical content-moderation-call-order shape.
  - **Negative — annotation bypass:** handler annotated with `@AllowContentWriteWithoutModeration("<reason>")` — passes despite missing moderate call. Reasons enumerated: `tombstone` (admin-driven content tombstone writes that replace user content with a fixed sentinel string), `admin_redaction` (Phase 3.5 admin chat-message redaction — content is replaced by a fixed system-controlled string), `seed` (Flyway seed migrations / test fixtures populating content directly).
  - **Negative — empty-string reason fails:** annotation with empty `""` reason still fires the rule (mirror precedent from `RateLimitTtlRule`'s `@AllowDailyTtlOverride` handling).
  - **Negative — test-source allowlist:** test files under `**/src/test/**` are allowlisted (mirror precedent: every existing rule has this allowlist).
  - **Negative — package-FQN fallback for synthetic test fixtures:** `id.nearyou.test.fixtures.*` package allowlisted (mirror canonical 3-rule precedent: `RedisHashTagRule.kt:115-120`, `RateLimitTtlRule.kt:92-100`, `BlockExclusionJoinRule.kt:176-186`).
  - **Negative — Repository own-content path allowlist:** mirror `RawFromPostsRule`'s admin-module + Repository own-content carve-out IF the rule's PSI walk doesn't already exclude these by virtue of the call-site location (verify during apply).
  - **Provider registration:** Kotest block `"rule registered in NearYouRuleSetProvider"` co-located inside the lint-test file (mirror the actual just-shipped pattern at [`OtelForbiddenAttributeLintTest.kt:807-812`](../../../lint/detekt-rules/src/test/kotlin/id/nearyou/lint/detekt/OtelForbiddenAttributeLintTest.kt) — there is NO separate `NearYouRuleSetProviderTest.kt` file; each rule's test file owns its own registration assertion).
- **MODIFY** `content-moderation-keyword-lists` capability — ADD a new requirement that the call-order contract is mechanically guarded at compile time by the `ContentWriteRequiresModerationRule` Detekt rule. The MODIFIED requirement is additive; no existing scenarios change. The closing prose around the existing "Moderator runs after length guard, before INSERT" scenario (currently at [`content-moderation-keyword-lists/spec.md:391`](../../../openspec/specs/content-moderation-keyword-lists/spec.md)) is amended to acknowledge the new compile-time defence layer alongside the existing test-time defence — matching the defense-in-depth phrasing introduced by `otel-attribute-lint-rule` (PR [#99](https://github.com/aditrioka/nearyou-id/pull/99)) on the `observability-otel-foundation` spec.
- DELETE the `content-write-moderation-detekt-rule` entry from [`FOLLOW_UPS.md`](../../../FOLLOW_UPS.md) at archive time — its scope is fully fulfilled by this change.

## Capabilities

### New Capabilities

(none — this change adds enforcement to an existing capability)

### Modified Capabilities

- `content-moderation-keyword-lists`: gains a lint-rule requirement that every write-path call site against `posts.content` / `post_replies.content` / `chat_messages.content` MUST be statically guarded by a preceding `TextModerator.moderate(...)` call OR an `@AllowContentWriteWithoutModeration("<reason>")` annotation with a non-blank reason in the `tombstone` / `admin_redaction` / `seed` enumeration.

## Impact

- **Code**: ~150 LOC for `ContentWriteRequiresModerationRule.kt` + KDoc; ~250 LOC for `ContentWriteRequiresModerationLintTest` covering the ~10 fixture scenarios; 1-line update to `NearYouRuleSetProvider`; 1-line addition for the `@AllowContentWriteWithoutModeration` annotation in `:core:domain` (or `:lint:detekt-rules` per the project annotation pattern — verify during apply).
- **Schema / APIs / Dependencies**: none.
- **Out of scope (explicit)**:
  - Layer 3 dispatch call-site enforcement. The `Layer3DispatchCallSiteTest.kt` already covers the post-INSERT Layer 3 dispatch surface (per `text-moderation-perspective-api-layer` capability); a Detekt rule for that surface would be a separate change.
  - Migration / SQL-file content writes. Detekt visits Kotlin PSI; `.sql` files (Flyway migrations) are reviewed in PR + the `seed` annotation reason covers the legitimate Flyway-driven content writes.
  - Mobile-side content-write enforcement. Mobile is currently DESIGN per [`openspec/project.md`](../../../openspec/project.md) § Mobile + Admin Status; once mobile content-creation paths land, the same rule extends to whatever surface they introduce (likely none — mobile calls the backend API, doesn't write content directly).
  - Detection of the Layer 3 dispatch failure-mode (which would require call-graph analysis Detekt cannot do without type-resolution mode). The runtime structured-log + integration-test sentinel scenarios from `text-moderation-perspective-api-layer` remain the canonical defence for that surface.
  - Cross-rule composition with `BlockExclusionJoinRule` or `RawFromPostsRule`. The new rule fires on the WRITE site; the existing rules fire on the READ site. They're orthogonal; no composition test required.
  - Coordinating rule-list ordering with [PR #100](https://github.com/aditrioka/nearyou-id/pull/100) (`rate-limit-ip-axis-method-lint-rule`) which is also adding the 8th rule. Whichever change merges to `main` second will need to rebase its `NearYouRuleSetProvider` edit to position the new entry as the 9th. Trivial conflict resolution.
