## Why

V5–V9 shipped the engagement primitives (blocks, follows, likes, replies, reports + auto-hide) but none of them emit a signal to the user on the receiving end of the interaction. `docs/02-Product.md:256` and `docs/08-Roadmap-Risk.md:152` both specify a DB-backed in-app notifications feed as the Phase 2 capability that turns those one-shot writes into a persistent per-user feed. The canonical schema, full event-type catalog, indexes, and retention policy are already decided at `docs/05-Implementation.md:820–867`. V10 is the natural next step: it introduces the `notifications` table and wires the four already-shipped engagement write-paths (like, reply, follow, 3-reporter auto-hide) to emit rows into it, plus the read endpoints the mobile feed will consume. FCM push dispatch is a separate change (Phase 2 item 7); the DB row is the source of truth per `docs/05-Implementation.md:849`.

## What Changes

- **V10 migration `V10__notifications.sql`** verbatim-aligned with `docs/05-Implementation.md:820–844` — adds one table:
  - `notifications (id UUID PK DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, type VARCHAR(48) NOT NULL CHECK (type IN (...13 values...)), actor_user_id UUID REFERENCES users(id) ON DELETE SET NULL, target_type VARCHAR(16), target_id UUID, body_data JSONB, created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(), read_at TIMESTAMPTZ)` + 2 indexes (`notifications_user_unread_idx` partial ON `(user_id, created_at DESC) WHERE read_at IS NULL`, `notifications_user_all_idx` ON `(user_id, created_at DESC)`).
  - **Full-enum-at-V10 convention (continues V9 precedent)**: the `type` CHECK enum ships with all 13 values from the `docs/05-Implementation.md:826–832` catalog even though V10 only writes 4 of them (`post_liked`, `post_replied`, `followed`, `post_auto_hidden`). The 9 reserved values (`chat_message`, `subscription_billing_issue`, `subscription_expired`, `account_action_applied`, `data_export_ready`, `chat_message_redacted`, `privacy_flip_warning`, `username_release_scheduled`, `apple_relay_email_changed`) are emitted by future features that don't need an enum-widening migration. Same pattern as V9's `moderation_queue.trigger` 7-value enum reserving future triggers.
  - **`actor_user_id` NULL convention**: 9 of the 13 event types are system-originated (subscription, admin action, data export, privacy flip, username release, Apple S2S, auto-hide) and have `actor_user_id = NULL` by design; only the 4 social events (`post_liked`, `post_replied`, `followed`, `chat_message`) carry an actor. The FK uses `ON DELETE SET NULL` (not CASCADE) because when an actor is hard-deleted we want the recipient's historical feed intact (a "someone liked your post" entry staying visible with the actor nulled out is the documented UX; the render layer shows "a deleted user" — handled on the client).
- **Write-path integration** — four already-shipped endpoints each gain a notification-emit side-effect, all in the same DB transaction as their primary write:
  - **Like** (`post-likes` V7): after the `post_likes` INSERT succeeds, if the post's `author_id != liker.id` AND the post author has not blocked the liker (UNIQUE `(blocker_id, blocked_id)` on `user_blocks`) AND the liker has not blocked the post author, INSERT a `post_liked` row for the post author with `actor_user_id = liker.id`, `target_type = 'post'`, `target_id = post.id`, `body_data = {"post_excerpt": <first 80 code points of post.content>}`.
  - **Reply** (`post-replies` V8): after the `post_replies` INSERT succeeds, if `parent_post.author_id != replier.id` AND no bidirectional block, INSERT a `post_replied` row for the parent post author with `actor_user_id = replier.id`, `target_type = 'post'`, `target_id = parent_post.id`, `body_data = {"reply_id": <reply uuid>, "reply_excerpt": <first 80 code points of reply.content>}`.
  - **Follow** (`follow-system` V6): after the `follows` INSERT succeeds, if no bidirectional block, INSERT a `followed` row for the followee with `actor_user_id = follower.id`, `target_type = NULL`, `target_id = NULL`, `body_data = {}`.
  - **Auto-hide** (`reports` V9): inside the V9 auto-hide transaction, after the `is_auto_hidden = TRUE` flip on `posts` or `post_replies`, INSERT a `post_auto_hidden` row for the target author with `actor_user_id = NULL`, `target_type = 'post' | 'reply'`, `target_id = <target uuid>`, `body_data = {"reason": "auto_hide_3_reports"}`. This lands the `// TODO: notifications-api-v??` stub V9 left behind.
- **Block-suppression rule** per `docs/02-Product.md:235` is enforced at write time, not at read time. Rationale: `docs/02-Product.md:235` says "Notifications from a blocked user: suppressed". Suppressing at write-time is cheaper (no row written, no 90-day retention storage, partial-index stays lean), bidirectional (covers both "I blocked them" and "they blocked me" symmetrically), and idempotent (a later unblock does NOT resurrect missed notifications — matching the spirit of the block feature that block is a forward-looking visibility boundary). The read path is consequently block-exclusion-free; the actor is already proven unblocked at emit time.
- **Self-action suppression**: `post_liked` / `post_replied` on own post, `followed` when the DB's CHECK on `follows` already rejects self-follow (defense-in-depth, the notification write-path mirrors the schema invariant). Centralized in a `NotificationEmitter.shouldEmit(actor, recipient)` helper so future emitters inherit the rule.
- **Read endpoints** — four new routes under `/api/v1/notifications`, all Bearer-JWT authenticated via the existing `auth-jwt` plugin:
  - `GET /api/v1/notifications?cursor=<created_at>&limit=<1..50>&unread_only=<bool>` — paginated list for the caller; uses `notifications_user_all_idx` or the partial `_unread_idx` depending on `unread_only`; returns `{items: [...], next_cursor: <ISO8601 or null>}`. Default `limit = 20`, max `50`.
  - `GET /api/v1/notifications/unread-count` — returns `{unread_count: <int>}`; uses the partial index for a cheap index-only scan.
  - `PATCH /api/v1/notifications/:id/read` — sets `read_at = NOW()` on the caller's row with `id = :id AND read_at IS NULL`; 404 if not found / already read / not owned; 200 no body.
  - `PATCH /api/v1/notifications/read-all` — sets `read_at = NOW()` on all rows `WHERE user_id = :caller AND read_at IS NULL`; returns `{marked: <int>}`.
- **Zero read-path changes to existing endpoints**. Timeline / reply-list / likes / follows / reports endpoints are untouched. Notifications is a purely additive surface.
- **Detekt `RawFromPostsRule` + `BlockExclusionJoinRule` both unmodified**. The write-path emitters JOIN on `user_blocks` (block-exclusion check is already their own query), and the read path touches only the `notifications` table (no `posts` / `users` / `post_replies` / `chat_messages` surface). No new exception entries needed.

## Capabilities

### New Capabilities
- `in-app-notifications`: DB-persisted per-user notification feed — `notifications` table schema + 2 indexes + 13-value `type` CHECK enum + FK edges (`user_id` CASCADE, `actor_user_id` SET NULL); write-path emitters with block-suppression and self-action suppression (4 types shipped: `post_liked`, `post_replied`, `followed`, `post_auto_hidden`); read endpoints (`GET /notifications`, `GET /notifications/unread-count`, `PATCH /notifications/:id/read`, `PATCH /notifications/read-all`) with JWT auth, cursor pagination, per-caller ownership filter; 90-day retention documented (worker out of scope); FCM hook point interface seam (no dispatch).

### Modified Capabilities
- `users-schema`: V10 adds two outgoing FK edges — `notifications.user_id → users(id) ON DELETE CASCADE` (recipient's feed wiped on hard-delete; matches the 90-day retention policy and the tombstone worker's existing behavior on other per-user tables) and `notifications.actor_user_id → users(id) ON DELETE SET NULL` (actor churn preserves recipient's feed with the actor nulled; distinct from CASCADE to match UX intent at `docs/05-Implementation.md:833`).
- `post-likes`: behavioral addition — successful like on another user's post now emits a `post_liked` notification for the post author within the same DB transaction, subject to block-suppression and self-action suppression. No change to existing like / unlike response shapes or to the `liked_by_viewer` field on timelines. Failure to emit a notification (DB error) MUST roll back the like itself (transactional coupling; no "like succeeded but notification lost" state).
- `post-replies`: behavioral addition — successful reply to another user's post now emits a `post_replied` notification for the parent-post author, transactional with the `post_replies` INSERT, block-suppressed, self-action-suppressed. No change to `POST /api/v1/posts/{id}/replies` or `GET /api/v1/posts/{id}/replies` response shapes.
- `follow-system`: behavioral addition — successful follow now emits a `followed` notification for the followee, transactional with the `follows` INSERT, block-suppressed. `DELETE` (unfollow) does NOT emit a counter-notification (no `unfollowed` type in the enum; matches `docs/05-Implementation.md:826` catalog). No change to `POST/DELETE /api/v1/users/:id/follow` response shapes.
- `reports`: behavioral addition — the existing V9 auto-hide transaction now also inserts a `post_auto_hidden` notification row for the target author (post or reply), resolving the V9 `// TODO: notifications-api-v??` stub. No change to the `POST /api/v1/reports` response shape or to the auto-hide threshold logic itself. Chat-message and user-target reports do NOT emit notifications (no `is_auto_hidden` flip on those target types; the `post_auto_hidden` type is specifically about content hiding).
- `migration-pipeline`: record V10; first migration to introduce a partial-index on a `read_at IS NULL` predicate (distinct from the plain B-tree pattern used for `username_history` and `csam_archive` per `docs/08-Roadmap-Risk.md:349` — `read_at IS NULL` is immutable, unlike `> NOW()` predicates, so the partial index is valid). Document this as the second valid partial-index pattern in the pipeline alongside the precedent from V1 `posts` partial indexes.
- `user-blocking`: behavioral note — block-suppression on notifications is write-time, not read-time. Unblocking a user does NOT resurrect notifications that were suppressed during the block window. No change to `POST/DELETE /api/v1/blocks` response shapes or to `GET /api/v1/blocks` list contract.

## Impact

- **Code**:
  - New: `backend/ktor/src/main/kotlin/id/nearyou/app/notifications/NotificationService.kt`, `NotificationEmitter.kt`, `NotificationRoutes.kt`, `NotificationController.kt`, `NotificationRepository.kt` (interface in `:core:data`) + `JdbcNotificationRepository.kt` (in `:infra:supabase`).
  - Touched: `LikeService`, `ReplyService`, `FollowService`, `ReportService` each gain a `NotificationEmitter` dependency + a single transactional emit call at the documented seam. No other changes to those services' logic.
  - New DTOs: `NotificationDto`, `NotificationListResponse`, `UnreadCountResponse`, `ReadAllResponse`, kotlinx.serialization.
  - No Detekt rule changes (no raw table reads in new code; emitters' block-check is a dedicated `user_blocks` query).
- **Schema**: new `backend/ktor/src/main/resources/db/migration/V10__notifications.sql`. V1–V9 immutable. Adds 1 table, 2 indexes, 13-value CHECK enum on `type`, 2 FKs (`user_id` CASCADE, `actor_user_id` SET NULL).
- **APIs**: 4 new endpoints (`GET /notifications`, `GET /notifications/unread-count`, `PATCH /notifications/:id/read`, `PATCH /notifications/read-all`). Zero changes to existing endpoints' response shapes.
- **Dependencies**: none new. PostgreSQL, Flyway, Ktor JWT, JDBC — all wired since Phase 1.
- **Observability**: structured log + Sentry breadcrumb on every successful emit (`event = "notification_emitted"`, `type`, `recipient_id`, `actor_id_or_null`, `suppressed_reason = null | "blocked" | "self_action"`); suppressed emissions log at DEBUG with the suppression reason to aid Phase 2-3 diagnostics. The Phase 3.5 dashboards at `docs/08-Roadmap-Risk.md:255` can pick this up.
- **Out of scope (explicit)**:
  - **FCM push dispatch** — Phase 2 item 7. This change ships a `NotificationDispatcher` interface with an in-app-only implementation; the FCM implementation lands in the follow-up change alongside `user_fcm_tokens` schema + `POST /api/v1/user/fcm-token` per `docs/08-Roadmap-Risk.md` Phase 1 item 18.
  - **`user_fcm_tokens` schema** — bundled with the FCM dispatch change.
  - **Notifications purge worker** (90-day retention enforcement) — Phase 3.5 admin-panel workers per `docs/08-Roadmap-Risk.md:242`. V10 documents the 90-day contract; actual DELETE is a Cloud Scheduler job shipped with the admin-panel change. The partial `_unread_idx` keeps the table lean in the meantime; the `_all_idx` degrades gracefully until the worker ships.
  - **In-app notifications for not-yet-shipped features** — `chat_message`, `subscription_billing_issue`, `subscription_expired`, `account_action_applied`, `data_export_ready`, `chat_message_redacted`, `privacy_flip_warning`, `username_release_scheduled`, `apple_relay_email_changed` are reserved enum values only. Each writer lands with its respective feature change.
  - **Rate limits** on `GET /notifications` / `PATCH /notifications/read-all` — separate rate-limit foundation change (Phase 1 item 24). This change inherits whatever default the route layer has (none, today); the read endpoints are cheap enough not to need per-endpoint limits until the rate-limit foundation lands.
  - **Mobile UI** (notifications list screen, unread badge, pull-to-refresh) — Phase 3 mobile work.
  - **Batching / collapsing** ("Alice, Bob, and 3 others liked your post" UX) — not in Phase 2 spec; deferred post-MVP. V10 writes one row per event.
  - **Read-path block-exclusion re-check** — deliberately omitted. Block-suppression is write-time; a post-emit block does NOT retroactively hide already-written notifications. Matches the forward-looking semantics of the block feature.
