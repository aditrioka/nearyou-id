## 3. Repository (upsert SQL)

- [ ] 3.1 Create `backend/ktor/src/main/kotlin/id/nearyou/app/user/FcmTokenRepository.kt` with a single `suspend fun upsert(userId: UUID, platform: String, token: String, appVersion: String?): UpsertResult` method.
- [ ] 3.2 Define `UpsertResult` as a data class `data class UpsertResult(val created: Boolean, val userTokenCount: Long)` carrying both the insert-vs-update distinction (per design D6) AND the per-user row count (per design D3 tripwire). Boolean / data-class shape — align with existing project repository conventions (grep `data class .*Result` under `backend/ktor/src/main/kotlin/`).
- [ ] 3.3 Implement `upsert` via a single SQL statement: `INSERT INTO user_fcm_tokens (user_id, platform, token, app_version, last_seen_at) VALUES (?, ?, ?, ?, NOW()) ON CONFLICT (user_id, platform, token) DO UPDATE SET last_seen_at = NOW(), app_version = EXCLUDED.app_version RETURNING xmax = 0 AS created, (SELECT COUNT(*) FROM user_fcm_tokens WHERE user_id = $1) AS user_token_count`. The `xmax = 0` Postgres trick distinguishes insert from update (reliable for plain `INSERT ... ON CONFLICT` against a non-partitioned table on PG 9.5+; the `(SELECT COUNT(*) ...)` subquery is a single index scan on `user_fcm_tokens_user_idx` — cheap). See `design.md` D6 for the `xmax = 0` mechanism notes.
- [ ] 3.4 Use a JDBC `PreparedStatement` with parameter binding (NEVER string concatenation — SQL injection guard).
- [ ] 3.5 Wrap the JDBC call in `withContext(Dispatchers.IO) { ... }` per the existing repository pattern.

## 4. Route handler

- [ ] 4.1 Create `backend/ktor/src/main/kotlin/id/nearyou/app/user/FcmTokenRoutes.kt` exporting `fun Application.fcmTokenRoutes(repository: FcmTokenRepository)` (or `fun Route.fcmTokenRoutes(...)` — match the precedent set by [`engagement/LikeRoutes.kt:42`](backend/ktor/src/main/kotlin/id/nearyou/app/engagement/LikeRoutes.kt:42) which is `fun Application.likeRoutes(service: LikeService) { routing { authenticate(AUTH_PROVIDER_USER) { ... } } }`). Register `POST /api/v1/user/fcm-token` inside the `authenticate(AUTH_PROVIDER_USER) { ... }` block.
- [ ] 4.2 Inside the handler, extract the `UserPrincipal` via `call.principal<UserPrincipal>() ?: return@post call.respond(HttpStatusCode.Unauthorized)` (defensive — the auth middleware should already 401 before reaching here, but the principal-null check is a belt-and-suspenders).
- [ ] 4.3 Receive the body via `call.receive<FcmTokenRequest>()` wrapped in a `try/catch` for `BadRequestException` / `JsonConvertException` / `ContentTransformationException` / `MissingFieldException` → respond `400 {"error": "malformed_body"}`. The `MissingFieldException` catch covers the scenario where the request body deserializes but a non-nullable field (`token` or `platform`) is absent.
- [ ] 4.4 Validate `platform` ∈ {`android`, `ios`} (case-sensitive lowercase) → on failure respond `400 {"error": "invalid_platform"}`. Empty string OR mixed-case both hit this branch (after deserialization succeeds with a non-empty but invalid value).
- [ ] 4.5 Validate `token.trim().isNotEmpty()` → on failure respond `400 {"error": "empty_token"}`.
- [ ] 4.6 Validate `(appVersion?.length ?: 0) <= 64` → on failure respond `400 {"error": "app_version_too_long"}`.
- [ ] 4.7 Call `repository.upsert(userId, platform, token.trim(), appVersion)` and capture the `UpsertResult { created, userTokenCount }`.
- [ ] 4.8 Log a single structured INFO event with EXACTLY these fields and NO others: `event="fcm_token_registered"`, `user_id`, `platform`, `created` (boolean), `user_token_count` (Long). MUST NOT include the raw `token` value (per design D11 + spec § Validation errors § "WARN logs do NOT include the raw token value" — this INFO event is also bound by the same redaction rule). MUST NOT include the request body or the `Authorization` header. Match the structured-logging style in [`backend/ktor/src/main/kotlin/id/nearyou/app/auth/`](backend/ktor/src/main/kotlin/id/nearyou/app/auth/) (grep `log.info` there for the SLF4J MDC pattern).
- [ ] 4.9 Respond `204 No Content` on success.
- [ ] 4.10 Wrap the whole handler in a top-level `try/catch (e: Exception)` for any unexpected throwable → log at WARN with redacted context (`log.warn("fcm_token_registration_failed user_id=$userId platform=$platform error_class=${e::class.simpleName}", e)` — does NOT include `token` value, does NOT include the request body) and respond `500 Internal Server Error` with no body. The catch MUST NOT include `BadRequestException` / `JsonConvertException` / `ContentTransformationException` / `MissingFieldException` — those are handled in 4.3. Also catch `PSQLException` for the DB-CHECK violation paths (token/app_version length): map to the corresponding `400` code if the constraint name is identifiable (e.g., `user_fcm_tokens_token_check` → `app_version_too_long` is wrong; the error is on token length — so respond `400 "empty_token"` if the bound is exceeded, given the only non-empty-token-failure mode at the DB is exceeding the upper bound; OR respond `500` if the constraint can't be cleanly mapped). Implementer's judgment, but err toward 500 if mapping is fragile — the app-level validators in 4.5/4.6 should already catch these cases; the DB CHECK firing means a bug in the validator.
